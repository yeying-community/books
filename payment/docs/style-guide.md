# Payment 文档写作规范（v0.1）

## 1. 表达规则

- 一段只讲一个中心意思。
- 先给结论，再给原因，再给例子。
- 能用短句不用长句，避免并列 5 层逻辑。
- 概念定义必须给边界条件（适用范围、不适用范围）。

## 2. 标题规则

- `H1`: `一、二、三...` 的章级结构，表达主题问题。
- `H2`: 分块拆解，表达子问题。
- `H3`: 实操项或分类项，避免抽象口号。
- 标题尽量“问题化”，例如：`为什么授权成功后仍可能拒付？`

## 3. 术语规则

- 首次出现：中文 + 英文 + 缩写，例如：`支付服务商（Payment Service Provider, PSP）`。
- 第二次开始统一用一种叫法，不再反复切换。
- 相似概念必须并列对照：`Authorization vs Clearing vs Settlement`。

## 4. 证据标签（建议）

为每个关键结论加标签，读者能快速判断可信度来源：

- `[FACT]` 客观事实（规则、标准、公开机制）
- `[RULE]` 组织规则或监管要求
- `[EXAMPLE]` 业务案例或模拟场景
- `[OPINION]` 作者判断与经验结论

## 5. 图表规则

- 每章至少 1 张总览图（关系图或时序图）。
- 每张图必须配三行说明：输入、处理、输出。
- 表格优先用于“对照”和“状态变化”，不要把段落塞进表格。

## 6. 常见反模式（避免）

- 只讲流程，不讲状态与责任归属。
- 只讲接口，不讲资金与账务落地。
- 只讲名词，不讲业务场景。
- 同一章节反复定义同一术语。
- 结尾没有总结和下一章衔接。

## 7. 建议的章节收尾格式

- 本章你应该记住：
- 实操中最容易踩的坑：
- 下一章将回答：
