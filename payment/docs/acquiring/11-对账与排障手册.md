## 3分钟速读（入门优先）

- “支付成功未到账”必须按交易层、清算层、结算层分层定位。
- 对账目的是确认一致性，排障目的是快速定位责任层级并止损。
- 差异分类字典和标准工单模板，是减少跨团队扯皮的关键。

## 一、本章要解决的问题

- 问题 1：出现“支付成功但未到账”时，如何快速定位问题层级？
- 问题 2：对账应该按哪些层次和粒度执行？
- 问题 3：如何建立标准化排障流程，减少跨团队扯皮？

## 三、分层对账模型

### 3.1 交易层对账

- 核对对象：支付请求、授权、Capture、回调。
- 核对目标：确认业务事件完整性与幂等一致性。

### 3.2 清算层对账

- 核对对象：平台 Capture 台账 vs 渠道清算结果。
- 核对目标：确认账务事实一致，识别漏清、重清、错清。

### 3.3 结算层对账

- 核对对象：应付商户、已出款、银行回执、到账结果。
- 核对目标：确认资金交收闭环与SLA达成情况。

## 四、标准排障路径（按“支付成功未到账”）

1. 查交易状态：是否 `AUTHORIZED/CAPTURED`。
2. 查清算状态：是否 `CLEARED`，金额币种是否一致。
3. 查结算状态：是否进入结算批次，是否被准备金或规则冻结。
4. 查出款状态：是否已出款，银行回执是否成功。
5. 查到账状态：是否在途，是否跨境中转延迟。
6. 结论归类：交易问题 / 清算问题 / 结算问题 / 银行问题。

## 五、差异分类字典（建议首版）

| 差异类型 | 典型现象 | 优先级 | 默认Owner |
|---|---|---|---|
| 漏单 | 平台有、通道无 | P0 | 交易技术 |
| 重单 | 同一交易多次入账 | P0 | 清算技术 |
| 金额差异 | 授权与清算不一致 | P1 | 清算运营 |
| 币种差异 | 交易币种与清算币种不一致 | P1 | 财务运营 |
| 结算延迟 | 超过SLA未出款 | P1 | 结算团队 |
| 银行退票 | 出款后失败回退 | P1 | 资金运营 |

## 六、故障分级与升级机制

- `P0`：资金安全风险、批量漏账、批量重复出款。
- `P1`：大面积延迟到账、核心通道异常。
- `P2`：局部商户或局部国家异常。

升级规则（建议）：

- P0：15 分钟内拉群 + 30 分钟内临时止损方案。
- P1：30 分钟内定位层级 + 2 小时内修复计划。
- P2：当天闭环处理并纳入周复盘。

## 七、排障工单模板（可复制）

```text
工单标题：{国家}/{通道}/{异常类型}/{日期}
交易号：{payment_id}
异常现象：{一句话}
当前状态：{交易/账务/资金状态}
影响范围：{笔数/金额/商户数}
初步判断层级：{交易|清算|结算|银行}
临时止损动作：{已执行动作}
责任人：{owner}
下一更新时间：{timestamp}
```

## 新手最容易错的 3 件事

1. 只看单笔交易日志，不做分层对账，导致问题定位反复回滚。
2. 没有统一差异分类字典，不同团队对同一问题定义不一致。
3. 工单信息不完整（影响范围、下一更新时间缺失），升级效率低。

## 八、Checklist

- [ ] 是否按交易/清算/结算三层做日常对账
- [ ] 是否有标准化差异分类字典
- [ ] 是否有故障分级与升级机制
- [ ] 是否有可复用工单模板与复盘模板

## 九、本章总结

- 对账本质是“确认一致性”，排障本质是“快速定位层级”。
- 没有分层模型，问题会在跨团队中长期悬而未决。
- 标准化字典和模板可以显著缩短处理时长。
