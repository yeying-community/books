## 一、本章要解决的问题

- 问题 1：如何把“写文档骨架”自动化，减少重复劳动？
- 问题 2：生成器输入字段、输出结构、覆盖策略如何统一？
- 问题 3：如何避免自动生成内容与现有文档规范冲突？

## 四、设计目标

- 标准化：统一章节骨架与元信息。
- 可控性：可配置生成范围和覆盖策略。
- 可追溯：每次生成都记录输入参数与版本。
- 可兼容：不破坏手工编写内容。

## 五、输入规范（摘要）

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `doc_id` | string | 是 | 文档编号，如 `21` |
| `title` | string | 是 | 文档标题 |
| `owner` | string | 是 | 负责人 |
| `version` | string | 是 | 版本号，如 `v1.2` |
| `audience` | array | 否 | 目标读者角色 |
| `sections` | array | 是 | 要生成的章节块 |
| `template_type` | string | 是 | `chapter` / `template` / `guide` |

## 六、输出规范（摘要）

- 输出路径：`docs/acquiring/{doc_id}-{title}.md`。
- 头部元信息：版本、更新时间、作者、审核人。
- 主体结构：按模板类型生成固定章节顺序。
- 链接规范：仅生成相对路径 markdown 链接。

## 七、覆盖与合并策略

1. 默认模式：`safe_append`（仅补齐缺失块，不覆盖已写内容）。
2. 强制模式：`force_replace`（覆盖指定块，必须提供备份开关）。
3. 冲突策略：同名章节存在时生成 `TODO-MERGE` 标记。
4. 审计记录：每次生成写入生成日志（时间、输入、输出文件）。

## 八、校验规则（强制）

- 结构校验：生成后必须包含最小章节集合。
- 链接校验：生成后必须通过 markdown 链接检查。
- 术语校验：关键术语必须可在术语表中检索到。
- 图谱校验：若包含 Mermaid，必须语法可渲染。

## 九、最小上线计划（建议）

1. 先实现 `chapter` 类型骨架生成。
2. 再实现模板库索引自动生成。
3. 最后接入 CI 检查（结构+链接+术语）。

## 十、本章总结

- 生成器是“提效工具”，不是“替代思考”。
- 先约束输入与输出，再做自动化。
- 有覆盖策略和审计日志，才能安全落地。
