## 四、审计对象与范围（强制）

1. 发布触发：触发人、时间、版本、环境、变更单号。
2. 门禁判定：规则版本、校验结果、阻断原因、阈值参数。
3. 审批过程：审批节点、审批人、结论、备注、时间戳。
4. 执行结果：发布成功/失败、回滚动作、恢复验证结果。
5. 通知闭环：告警渠道、送达状态、确认时间。

## 五、证据链模型

```mermaid
flowchart LR
  A[发布触发记录] --> B[门禁校验结果]
  B --> C[审批轨迹]
  C --> D[发布执行日志]
  D --> E[回滚/恢复记录]
  E --> F[监管报表汇总]
```

图说明：

- 输入：发布元数据、门禁结果、审批记录、运行日志。
- 处理：证据采集、关联归档、口径汇总、报表生成。
- 输出：可追溯审计链路和监管报表。

## 六、留痕字段最小集合

| 维度 | 字段示例 | 要求 |
|---|---|---|
| 身份 | `actor_id`、`actor_role` | 必须可追溯到具体责任人 |
| 时间 | `trigger_time`、`approve_time` | 使用统一时区与格式 |
| 变更 | `release_version`、`rule_version` | 版本字段不可缺失 |
| 结果 | `gate_decision`、`rollback_result` | 需有明确枚举值 |
| 关联 | `ticket_id`、`incident_id` | 可关联工单与事故复盘 |

## 七、留存策略（建议）

| 数据类型 | 最低留存周期 | 存储建议 |
|---|---|---|
| 审批轨迹 | 18 个月 | 冷热分层 + 不可篡改存档 |
| 发布日志 | 12 个月 | 主存储 + 每日增量备份 |
| 回滚与恢复记录 | 24 个月 | 高优先级长期归档 |
| 监管报表 | 按监管要求（通常 24 个月+） | 版本化快照与签名 |

## 八、监管报表口径治理

1. 单一口径：同一指标在审计和报表中定义一致。
2. 版本口径：每次报表字段变更必须记录版本号和生效日期。
3. 可复算：报表结果可从原始审计字段重算得到。
4. 可解释：关键异常项必须有备注和责任人。

## 九、常见审计风险

1. 字段缺失：关键字段未采集，导致证据链断裂。
2. 时间混乱：多时区未统一，审计时间线错位。
3. 口径漂移：指标定义变更未公告，报表不可比。
4. 归档失效：日志存在但无法检索或无法关联工单。

## 十一、本章总结

- 没有审计留痕的自动发布，本质上不可治理。
- 证据链完整性比“日志数量”更关键。
- 报表口径标准化是跨团队协同和监管应对的基础。
