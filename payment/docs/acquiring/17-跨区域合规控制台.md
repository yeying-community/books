## 四、控制台目标与边界

### 4.1 核心目标

- 统一规则视图：各国家合规差异可结构化查询。
- 统一变更路径：规则发布必须经审批、可审计、可回滚。
- 统一风险信号：合规告警可与风控、路由、结算联动。
- 统一审计证据：审计期间可快速导出变更与执行证据。

### 4.2 系统边界

- 控制台负责：规则管理、审批流、留痕、告警编排。
- 业务系统负责：执行规则与回传执行结果。
- 法务/合规负责：规则合法性审查与例外审批。

## 五、推荐控制台能力（MVP）

1. 国家规则矩阵：支付方式、认证、KYC、数据限制、禁限行业。
2. 规则版本中心：按国家/产品维护版本，支持灰度与回滚。
3. 审批工作流：产品 -> 风控 -> 合规 -> 法务 -> 发布。
4. 执行监控：规则命中率、违规命中、例外审批通过率。
5. 审计导出：按时间窗口导出规则版本、审批链和执行证据。

## 六、统一治理规则（强制）

### 6.1 规则版本化

- 所有规则必须具备 `rule_id`、`rule_version`、`effective_at`。
- 任何规则变更必须关联审批单号与发布单号。
- 历史版本不可覆盖，只能新增版本并失效旧版本。

### 6.2 审批与发布

- 高风险规则（认证、KYC、资金限制）必须四级审批。
- 低风险规则（文案、告警阈值）可简化审批但必须留痕。
- 发布必须具备回滚标签与影响范围评估。

### 6.3 例外管理

- 例外必须具备有效期、责任人和到期提醒。
- 到期未续签例外自动失效并恢复默认规则。

## 七、核心指标（建议）

| 指标 | 定义 | 目标方向 |
|---|---|---|
| 规则覆盖率 | 已结构化国家规则 / 总覆盖国家 | 上升 |
| 审批及时率 | SLA内完成审批 / 总审批单 | 上升 |
| 发布回滚率 | 回滚发布次数 / 总发布次数 | 下降 |
| 审计取证时长 | 审计请求到证据导出时长 | 下降 |
| 例外逾期率 | 逾期未处理例外 / 总例外 | 下降 |

## 九、本章总结

- 跨区域合规能力本质是“规则工程化”，不是“文档堆叠”。
- 没有版本化和审批留痕，合规变更不可控。
- 控制台价值在于统一治理和快速应对监管变化。
