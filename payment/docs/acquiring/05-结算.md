## 3分钟速读（入门优先）

- 结算回答的是“钱什么时候、按什么规则给到商户”。
- 清算确认账务，结算完成资金交收，两者不能互相替代。
- 结算系统要同时支持增长（到账体验）和风险控制（准备金、延迟出款）。

## 一、本章要解决的问题

- 问题 1：结算（Settlement）与清算（Clearing）的本质差异是什么？
- 问题 2：为什么商户到账常晚于用户支付时间？
- 问题 3：结算系统如何同时支持增长与风险控制？

## 四、核心概念定义

### 4.1 结算的三层对象

- 成员层结算：Network 与成员机构间交收。
- 平台层结算：平台内部清分、费用归集、准备金处理。
- 商户层结算：面向商户的最终应付与出款。

### 4.2 结算不是打款动作

- 定义：打款只是执行动作，结算是规则与账务体系。
- 边界：结算系统必须可解释每一笔应付来源。
- 常见误解：把结算系统等同“出款接口”。

### 4.3 银行间资金交收视角：Nostro / Vostro

- 背景：跨境资金交收并不总是“每来一笔就实时跨境汇一笔”。
- 常见做法：成员机构会在代理行预存头寸，后续优先用既有头寸完成资金交收。
- `Nostro`：本机构存放在他行的资金，在本机构账上通常体现为资产。
- `Vostro`：他行为本机构保管的该笔资金，在他行账上通常体现为负债。
- 关键关系：`Nostro` 与 `Vostro` 是同一笔预存头寸在双方账上的镜像记录，金额对应、会计属性相反。
- 边界提醒：这是银行间资金交收视角，不等同于支付平台全部结算模型；平台侧仍需独立处理清分、费率、准备金与商户应付。

## 五、主流程拆解

### 5.1 阶段 1：结算批次与规则计算

- 参与方：结算系统、财务系统、费率系统。
- 关键输入：清算净额、费率模板、结算周期、币种规则。
- 核心动作：汇总应收应付并生成结算批次。
- 关键输出：结算应付台账。

### 5.2 阶段 2：准备金与风险联动

- 参与方：风控、结算、商户运营。
- 关键输入：商户风险等级、争议率、退款率、行业参数。
- 核心动作：冻结/释放准备金，动态调整结算周期与比例。
- 关键输出：风险调整后的可出款金额。

### 5.3 阶段 3：出款与到账回执

- 参与方：出款系统、银行通道、对账系统。
- 关键输入：收款账户信息、出款指令、银行返回码。
- 核心动作：执行出款、接收回执、异常重试与人工处理。
- 关键输出：到账状态、失败原因、重试记录。

### 5.4 阶段 4：代理行头寸与同业清算衔接

- 参与方：付款行、代理行/收款行、总账/同业账务系统。
- 关键输入：客户付款指令、代理行头寸余额、同业往来科目、清算指令。
- 核心动作：先确认客户侧扣账，再通过 `应付同业`/`待清算` 等中间科目衔接同业清算，最终扣减付款行 `Nostro`，同步减少收款行 `Vostro`。
- 关键输出：客户账、同业账、头寸账三层一致的资金交收结果。

最小示意：

1. 付款行先扣减付款客户余额，并挂入 `应付同业` 或 `待清算`。
2. 同业清算执行时，付款行减少在代理行的 `Nostro` 资产。
3. 收款行同步减少对应 `Vostro` 负债，并增加收款客户存款。
4. 整个过程结束后，客户账变化、同业往来变化、代理行头寸变化必须能够逐层对上。

这样设计的原因：

- 便于把“客户交易”和“同业清算”分开建模，降低账务解释难度。
- 便于满足监管、审计和统计口径要求，单独追踪清算业务动账。
- 便于在头寸不足、清算延迟、人工回补时保留清晰的中间状态。

## 六、常见异常与误区

### 6.1 可结算金额与商户预期不一致

- 现象：商户认为应到账金额高于实际出款金额。
- 根因：费用、准备金、争议预留、汇率处理理解不一致。
- 排查路径：按结算明细逐项解释 -> 对齐合同条款 -> 输出可读账单。

### 6.2 出款成功但银行未到账

- 现象：平台显示出款完成，商户未收到款。
- 根因：银行处理中、收款账户问题、跨境中转时延。
- 排查路径：核对银行回执状态 -> 核对账户信息 -> 走失败回退或重发流程。

## 七、实战案例

案例背景：

- 地区：东南亚
- 支付方式：外卡 + 本地钱包
- 商户类型：平台型电商
- 关键约束：多币种结算与多主体清分

案例过程：

1. 平台按店铺维度清分，再按商户主体生成结算单。
2. 对高争议品类设置滚动准备金，降低资金回收风险。
3. 对账系统按批次输出“应付、已付、在途、失败”四态。

案例结论：

- 成功点：结算状态可观测，资金解释成本显著下降。
- 失败点：早期未独立建模“在途”状态，导致客服口径混乱。
- 可复用策略：结算要先“算清楚”，再“打出去”。

## 新手最容易错的 3 件事

1. 把出款接口当作结算系统，忽略费用、准备金和规则引擎。
2. 结算账单不可解释，导致商户反复质疑到账金额。
3. 只看出款成功率，不跟踪在途和退票回退状态。

## 八、Checklist

- [ ] 是否具备可配置结算周期与模板能力
- [ ] 是否支持准备金动态策略
- [ ] 是否有出款失败重试与人工兜底
- [ ] 是否有商户可理解的结算账单

## 九、本章总结

- 结算核心是资金确定性，不是单纯转账动作。
- 结算系统质量直接决定商户体验与风险敞口。
- 规则可解释性与状态可观测性同等重要。
