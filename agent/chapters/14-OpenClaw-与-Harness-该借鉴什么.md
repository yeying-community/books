# 第十四章：OpenClaw 与 Harness，该借鉴什么

如果说 `Claude Code` 给社区展示的是“本地代码代理可以做得多深”，那么 `OpenClaw` 和 `Harness` 给出的，是另外两种同样重要的启发：

- 如何把 Agent 做成一个更容易扩展、接入和传播的开源产品。
- 如何把看似分散的问题，重新放回统一的运行时框架里理解。

## 14.1 OpenClaw 真正有价值的不是神话，而是拆分方式

围绕 `OpenClaw` 的讨论里，最需要去噪的是各种夸张叙事。把它说成“万能数字员工”不准确，把它说成“自动化套壳”也不准确。

从工程上看，更接近事实的表述是：

`OpenClaw` 把聊天入口、Agent 编排、本地或远程执行环境，以及技能扩展机制拆成了一套更容易被开源社区理解和搭建的数字员工框架。

这个判断现在可以被一手材料进一步钉住。

官方仓库首页对它的定位非常直接：`Gateway` 只是控制平面，产品本身是那个能在你已有渠道里和你持续互动的 assistant。官方文档也把多 Agent 路由、技能、工作区、沙箱、会话和事件都放在同一套 Gateway 运行时里解释。

这意味着前面说的“入口、大脑、执行器、技能”并不是二手文章为了叙事方便随手总结出来的，而是官方产品结构本身就在强调的一种拆法。

这类拆分至少有三个现实价值：

第一，入口独立。微信、飞书、Telegram、网页、CLI 这些入口不是附属品，而是产品化的分发面。很多项目之所以火，不是因为“思考能力突然变强”，而是因为终于把入口接到了用户真实工作流里。

第二，执行器独立。把“能不能动手”单独收敛成执行层，有助于隔离权限、环境依赖和部署差异。

第三，技能独立。能力被封装成技能或插件后，生态扩展门槛会显著下降。这往往比继续堆系统 Prompt 更能推动社区传播。

因此，研究 `OpenClaw` 时，重点不该是“它会不会替代所有人”，而是“它如何把一套 Agent 产品拆成可组合部件”。

## 14.2 OpenClaw 给出的另一个重点是上下文工程

参考材料中关于 `OpenClaw` 最值得保留的一部分，是它把上下文工程拆成几层：

- 收集层：用户配置、工作区文档、对话历史、工具列表、长期记忆。
- 组装层：按照固定顺序和模型兼容要求，拼成最终上下文。
- 保护层：清理脏消息、做兼容修复、控制上下文长度、必要时触发压缩。

这类分层很有参考价值，因为它提示我们：

Agent 的上下文工程不是“把资料拼一拼”，而是一条持续运行的流水线。流水线里既有信息组织问题，也有质量控制问题，还有模型兼容和预算管理问题。

这和 `Claude Code` 的运行时设计其实并不冲突，只是呈现方式不同。前者更偏开源框架式暴露，后者更偏产品级内聚实现。

如果进一步往工程里走，`OpenClaw` 这种拆法至少回答了三个非常现实的问题：

第一，渠道接入如何不把系统主体拖乱。

很多数字员工类系统真正先火起来的，不是因为推理能力突然跃升，而是因为它们把入口接到了真实工作流里。但入口一旦变多，系统就很容易被渠道细节反向绑架。把入口层单独收束成 `Gateway` 或 adapter，本质上是在保护主体运行时不被渠道协议污染。

第二，执行面如何不和推理面缠死。

一个工程成熟度较低的 Agent，经常会把模型判断和具体执行写在一团流程里。这样一来，只要执行环境变化，推理逻辑就会跟着重写。`OpenClaw` 这种执行器独立的拆法，价值就在于承认“思考”和“动手”是两种需要分开演进的能力。

第三，能力扩展如何不总靠改核心代码。

技能系统的存在，意味着扩展能力可以通过独立模块进入系统，而不必每次都深入到主循环里改判断逻辑。这对开源社区尤其重要，因为它把“贡献新能力”的门槛从“理解整个项目架构”降低成“补一个边界清楚的能力单元”。

这三点合在一起，构成了 `OpenClaw` 对开源 Agent 社区最现实的价值：它不是把所有问题都解决了，而是把问题拆成了更多人能参与、能扩展、能替换的模块。

## 14.3 OpenClaw 为什么更像一个产品骨架，而不只是技术 Demo

很多开源 Agent 项目看起来功能不少，但读完代码后仍然像实验作品。`OpenClaw` 相对更值得注意的一点，是它更接近产品骨架。

这里的“产品骨架”不是指界面更完整，而是指它天然关心几类 demo 之外的问题：

- 用户是从哪里进入系统的。
- 多种入口怎样映射到统一消息格式。
- 能力如何以技能方式增长。
- 执行环境如何隔离。
- 多 Agent 或多角色协作如何进入框架，而不是每次临时拼装。

换句话说，`OpenClaw` 更像是在回答“如果这真是一套数字员工平台，系统该长成什么样”，而不只是回答“模型能不能调用几个工具”。

这点对写这本书很重要，因为它帮助我们把 Agent 分成两类不同对象：

- 一类更像具体产品实现，例如 `Claude Code`。
- 一类更像开源产品骨架，例如 `OpenClaw`。

两者都值得研究，但阅读重点不一样。前者适合学运行时细节，后者更适合学系统拆分与可扩展性。

## 14.4 从官方文档看，OpenClaw 的骨架到底落在哪些模块上

如果只看二手分析，很容易把 `OpenClaw` 理解成“接了一堆聊天渠道的 AI 壳子”。但从官方仓库 README 和文档结构反推，它的骨架至少明确落在下面几块。

### Gateway 是控制平面

官方仓库直接把 `Gateway` 描述成控制平面。这一点非常关键，因为它说明系统不是把“接消息”和“跑 Agent”混成一个黑盒，而是有意把会话、渠道、工具和事件收束在一层统一调度面里。

这个设计的价值在于：渠道再多，主体运行时仍然只有一个中心。否则每接一种渠道，系统就会多一套局部逻辑，最后很难维护。

### Workspace 是运行时现场

官方文档现在已经把工作区描述得更具体了：默认会围绕 `agents.defaults.workspace` 组织运行时现场，并把一组 bootstrap 文件直接注入上下文。

其中最核心的文件包括：

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`

除此之外，官方文档还明确列出了：

- `IDENTITY.md`
- `USER.md`
- `MEMORY.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md`

这里最值得注意的不是文件名本身，而是它暴露出的设计选择：`OpenClaw` 把身份、人格、用户画像、工具说明、局部记忆和运行时提示都显式文件化了。

这说明 `OpenClaw` 的上下文工程不是临时拼接，而是明确把工作区当成运行时现场来设计。也就是说，规则、人格、工具说明和局部工作资料都不是散落在系统边角里，而是通过工作区显式进入主循环。

### Skills 是一级扩展机制

官方文档对 `skills` 的描述也很清楚：既有安装在共享根目录下的技能，也有工作区级技能，还有按 agent 可见性做的 allowlist 配置。

这和很多项目把插件当成后挂能力不同。`OpenClaw` 的技能更接近运行时里的一级模块，它既是能力扩展方式，也是不同 agent 之间做能力裁剪的边界。

### Multi-Agent Routing 是隔离系统，不只是“多开几个 agent”

官方文档里关于多 Agent 的描述非常明确：一个 agent 不是一个聊天昵称，而是一个完整隔离的“脑”。

每个 agent 至少有自己的：

- workspace
- `agentDir`
- session store

并且通过 `bindings` 把不同渠道、账号或 peer 路由到不同 agent。

这意味着 `OpenClaw` 的多 Agent 不是临时起多个实例，而是把“路由 + 状态隔离 + 身份隔离”正式做成了系统层能力。

另外，官方子 Agent 文档还给了一个很重要的补充：子 Agent 运行时默认使用更小的 prompt 形态，只注入 `AGENTS.md` 与 `TOOLS.md` 这类更偏任务和能力边界的内容，而不会把 `SOUL.md`、`IDENTITY.md`、`USER.md`、`HEARTBEAT.md`、`BOOTSTRAP.md` 全量带进去。

这说明它的多 Agent 设计并不是简单复制主 Agent 上下文，而是在做一种更克制的上下文裁剪。对子 Agent 来说，最优先的是任务边界和可用能力，而不是完整人格包装。

## 14.5 OpenClaw 的多 Agent，重点是路由隔离，不是协作魔法

这也是官方文档帮助我们纠正二手印象的一个地方。

很多人一看到多 Agent，就自然会联想到“几个 agent 互相讨论、协作、共识”。但从官方一手材料看，`OpenClaw` 当前最明确、最稳定的重点其实是路由隔离：

- 不同渠道消息进来，怎么路由到不同 agent。
- 不同 agent 的工作区和状态怎么分开。
- 不同 agent 的技能和工具可见性怎么分开。
- 不同 agent 的认证资料怎么分开。

这件事的工程价值比“多 Agent 会不会互相协作”更基础，也更重要。因为很多真实系统首先需要的不是智能协作，而是把“家庭 agent”“工作 agent”“公开渠道 agent”“高权限 agent”先拆开，不让它们共享同一份上下文和同一套凭证。

从这个角度看，`OpenClaw` 的多 Agent 设计更像一套运行时隔离体系，而不是一套炫技式的群体智能系统。

## 14.6 官方安全模型进一步说明了它为什么像平台而不只是工具

官方仓库 README 里还有一个非常关键的信号：它把安全模型单独拿出来强调，而且明确写了默认情况下主会话工具运行在宿主机上。

同时，官方文档又提供了：

- `sandbox.mode`
- per-agent sandbox
- per-agent tool allow/deny
- 非主会话沙箱化

这些配置能力。

这说明 `OpenClaw` 的定位本身就不是“单一用途工具”，而是一个需要面对多渠道、多 agent、多权限层次的运行平台。否则它没有必要把安全模型、路由绑定、agent 目录、技能可见性和沙箱配置做成这样一套系统。

换句话说，`OpenClaw` 更值得学的地方，不是某一个功能点，而是它承认了一件事：一旦 Agent 进入真实环境，它迟早会从“一个助手”长成“一个平台”。

## 14.7 Harness 的真正贡献是把问题重新命名

`Harness` 相关材料之所以值得长期参考，不是因为它发明了所有机制，而是因为它帮社区把一些原本松散的问题重新放回统一框架。

最重要的重命名有三个：

第一，Agent 的稳定性来自 Harness，而不是来自模型。这句话听起来像口号，但它实际迫使我们把注意力从“模型多强”转向“系统如何运行”。

第二，上下文窗口不是记忆，而是工作台。这个说法直接把很多混淆问题切开了。

第三，工具、记忆、约束、反馈、恢复、压缩，都不是旁支模块，而是运行时层的核心职责。

一旦接受这套框架，很多争论会简单很多。比如：

- 为什么有些 Agent demo 很惊艳，但一进生产就崩。
- 为什么上下文窗口越大，很多团队反而越迷茫。
- 为什么工具越多，不代表系统越强，反而常常更乱。

因为真正缺的不是能力点，而是 Harness。

如果再往前走一步，`Harness` 的意义还不只是“换个说法”，而是强迫系统设计者把责任重新分配清楚。

过去很多团队喜欢把 Agent 失败统称为“模型不稳定”。但在 `Harness` 框架下，至少要继续追问：

- 是上下文组织不对，还是模型不行。
- 是工具契约不清，还是模型不会调。
- 是短期状态丢了，还是模型忘了。
- 是权限边界没画清，还是模型太激进。
- 是没有恢复机制，还是模型不会续跑。

这类追问的价值非常大，因为它会把很多原本模糊的失败现象，重新压回具体模块。

也正因为如此，`Harness` 对社区最重要的贡献之一，是提供了一种更可诊断的分析方式。它让“Agent 做不好”不再是一个笼统判断，而变成一组可以拆分定位的问题。

## 14.8 Harness 视角下，运行时到底应该承担什么

如果把 `Harness` 的视角进一步压实，可以把运行时的职责至少分成六块：

### 上下文治理

决定哪些信息进入工作台、哪些信息延迟加载、哪些信息需要被压缩或折叠。

### 工具路由

决定模型能接触哪些能力、每种能力如何描述、返回结果如何重新包装，以便继续进入下一轮推理。

### 状态续航

决定工作记忆、短期状态和长期记忆怎样协同，尤其是长任务如何不断线。

### 权限控制

决定系统是否有资格继续往下执行，而不只是“能不能点一下确认”。

### 恢复机制

决定系统在中断、失败、压缩之后，能否从一个可执行状态继续，而不是重新开始。

### 评测治理

决定系统如何知道自己到底是在变强，还是只是在制造更复杂的假象。

把这六块放在一起，就能看出 `Harness` 为什么适合作为理论框架。它不是替代具体实现，而是给具体实现提供了一张职责分布图。

## 14.9 把 Claude Code、OpenClaw、Harness 放在一起时，应该怎样读

到这里，可以把三者的阅读方式明确地区分开：

### 读 Claude Code

重点看一个高耦合产品怎样把运行时细节做深。

要看的是：

- Prompt 装配如何进入系统骨架。
- 工具契约如何写进执行面。
- 权限如何嵌入运行时。
- 压缩与恢复如何支撑长任务。

### 读 OpenClaw

重点看一个开源框架怎样把 Agent 产品骨架拆开。

要看的是：

- 入口怎样模块化。
- 执行器怎样独立。
- 技能怎样扩展。
- 多 Agent 和上下文流水线怎样纳入框架。

### 读 Harness

重点看如何把所有这些分散机制重新放回统一运行时理论。

要看的是：

- 哪些问题其实属于同一层职责。
- 哪些失败是系统设计问题，而不是模型问题。
- 一套生产级 Agent 运行时最少该承担哪些责任。

这三种阅读方式一旦分清，后面再加入新的项目，也更容易定位它到底更适合当产品样本、框架样本，还是方法论样本。

## 14.10 当前最值得沉淀的共同经验

把 `Claude Code`、`OpenClaw`、`Harness` 放在一起看，当前至少能沉淀出六条比较稳定的经验：

1. 先画主链路，再看局部技巧。用户输入如何进入系统，比功能列表更重要。
2. Prompt 必须被当成装配层管理，而不是静态文案。
3. 工具系统要提供契约和反馈，而不是只追求能力暴露。
4. 权限系统要进入运行时核心，而不是停留在确认弹窗。
5. 上下文、短期状态、长期记忆必须分层。
6. 入口、执行器、技能和记忆最好各自解耦，否则系统很难扩展。

这些经验未必只属于某一个项目，但它们已经足够构成我们继续写这本书的分析框架。

如果把这几条经验再往方法论方向压一步，还可以得到一层更实用的判断：

- 产品型 Agent，先看主链路是不是闭合。
- 框架型 Agent，先看模块边界是不是清晰。
- 生产型 Agent，先看运行时职责是不是完整。

这三种判断方法，基本可以覆盖当前社区大多数项目。

后续再往下写时，最值得继续深挖的四个主题是：

- Prompt 装配与缓存稳定性。
- 工具路由与行为约束。
- 权限、Hook、MCP、技能带来的攻击面。
- 长任务里的压缩、恢复和子 Agent 协作。

## 14.11 新增四篇架构设计资料，哪些值得吸收

这次补进来的四篇资料，主题分别是记忆系统、工具系统、执行循环和多 Agent 协作。把它们当作次级信源来读，整体是有价值的，但用途需要控制。

最值得吸收的部分主要有四类：

第一，问题拆分方式比较清楚。

例如它们会把记忆拆成“存什么、存在哪、怎么取、怎么管”，把多 Agent 拆成“角色分离、上下文隔离、通信协调、结果汇总”。这类拆法很适合拿来增强本书的方法论密度。

第二，对 `OpenClaw` 的产品骨架理解有帮助。

尤其是入口层、执行层、技能层和心跳/定时触发这类组织方式，确实能帮助读者更好地理解为什么它更像一个开源平台骨架，而不只是一个工具调用 demo。

第三，对 `Claude Code` 的几个重点模块提供了较强线索。

例如工具分级、按需加载外部工具定义、显式规划、自愈查询循环、子 Agent/团队协作模式，这些都值得保留为分析方向，但仍然应该尽量回到官方文档、源码或更直接的证据做核验。

第四，对 `Harness` 视角是有强化作用的。

它们不断提醒一件事：很多看似分散的问题，其实都属于运行时职责。这和前面理论篇对 `Harness` 的处理是一致的。

同时，也有两类地方需要明确降权：

- 具体数字、内部实现细节、实验性功能命名。
- 对某些项目的意图推断和安全结论。

这些内容可以作为线索，但不应该未经交叉验证就直接写成定论。

因此，我对这四篇资料的整体判断是：

- 作为架构拆分参考，有用。
- 作为方法论补充，有用。
- 作为一手事实来源，不够。
- 作为最终结论依据，不够。

把它们放进当前这本书里，最合理的位置不是单独开章节，而是作为“问题细化框架”的补强材料，分散吸收到理论篇和项目对照篇里。
