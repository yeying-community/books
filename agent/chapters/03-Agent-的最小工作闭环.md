# 第三章：Agent 的最小工作闭环

如果把所有产品形态、界面差异和包装术语都拿掉，一个 Agent 至少要有一个最小工作闭环。

这个闭环通常包含七个基本环节：

1. 目标：系统需要知道当前要完成什么。
2. 上下文：系统需要拿到与当前目标相关的规则、环境和状态。
3. 推理：模型基于目标和上下文判断下一步动作。
4. 工具：系统通过受约束接口接触外部世界。
5. 反馈：工具结果返回给模型，成为下一轮判断的输入。
6. 循环：系统可以连续推进，而不是一次回答后结束。
7. 终止：系统知道什么时候该结束、交还控制权或请求确认。

这七步缺任何一块，系统都很难真正长期运行。

例如：

- 没有上下文，就只能瞎猜。
- 没有工具，就只能停留在建议层。
- 没有反馈，就无法根据执行结果修正路径。
- 没有终止条件，就容易无限循环。

很多所谓“Agent demo”之所以只能演示几分钟，就是因为它们虽然有推理和工具，但闭环里的其他部分非常薄弱。

因此，后面分析任何项目时，都可以先问这七个问题：

- 目标从哪里来。
- 上下文如何装配。
- 推理依赖什么模型与规则。
- 工具边界如何定义。
- 反馈是否结构化。
- 循环如何控制。
- 终止条件如何设定。

这套最小闭环，就是后面整本书反复使用的第一把尺子。

## 3.1 为什么一定要先讲闭环

很多人讨论 Agent 时，习惯先从更显眼的东西开始：

- 用了什么模型。
- 接了多少工具。
- 能不能多 Agent。
- 支不支持 MCP、Skills、RAG、记忆库。

这些当然都重要，但如果没有闭环视角，讨论很快就会碎掉。你会知道系统“有什么”，却仍然不知道系统“是怎么工作的”。

闭环的价值就在这里。

它不是一个概念装饰，而是一种最小分析结构。无论一个 Agent 看起来多复杂，最后都必须回答一个朴素问题：

它怎样从目标出发，经过上下文、推理、工具、反馈和循环，最终抵达一个可停止的状态。

只要这个问题回答不清楚，系统能力堆得越多，越容易让人误把功能列表当成架构。

## 3.2 目标：没有目标，后面所有动作都只是漂移

闭环的第一步是目标。

这看起来过于 obvious，但很多系统恰恰在这里模糊不清。它们收到的是用户输入，却没有把输入转成一个足够稳定的任务目标。

目标不清晰会直接导致三个问题：

- 系统不知道自己该做到什么程度才算完成。
- 系统不知道当前步骤是在推进主线，还是只是在局部忙碌。
- 系统在长任务里很容易把最近一步误当成整体目标。

因此，一个真正可运行的 Agent，不只是“听懂用户在说什么”，还必须把用户输入转成一份可执行目标。

这份目标至少要隐含几个要素：

- 要解决什么问题。
- 到什么状态算完成。
- 是否允许多步推进。
- 哪些边界不能越过。

目标越清楚，后面每一轮判断越不容易漂移。

## 3.3 上下文：不是背景资料，而是当前工作现场

有了目标之后，系统下一步面对的不是“开始推理”，而是“当前该把什么带上工作台”。

这就是上下文的真正角色。

上下文不是所有背景信息的总和，而是当前这一轮推理真正可见的工作现场。它至少会包含：

- 当前任务目标。
- 相关规则和约束。
- 当前环境信息。
- 最近的关键历史。
- 当前可用工具。

上下文之所以关键，是因为模型不会从上下文外替你补这层现实。

如果上下文过少，系统会臆测；

如果上下文过多，系统会发散；

如果上下文组织混乱，系统即使“看见了所有东西”，也可能抓不住重点。

因此，在闭环里，上下文不是配料，而是决定这一轮推理质量的工作现场。

## 3.4 推理：推理不是答案生成，而是下一步决策

很多非工程讨论会把“推理”理解成模型输出看起来更聪明、更像思考的文字。

但放到 Agent 闭环里，推理更准确的作用是：基于当前目标和上下文，决定下一步该做什么。

这个“下一步”可能是：

- 直接回答。
- 先补一个问题。
- 调用某个工具。
- 先读文件再决定。
- 停下来请求确认。
- 发现已经可以结束。

也就是说，闭环里的推理并不是抽象思辨，而是控制流决策。

这也是为什么 Agent 比普通聊天系统更依赖推理稳定性。聊天系统推理得差一些，可能只是回答得不够好；Agent 推理得差一些，可能会直接把系统带到错误执行路径上。

## 3.5 工具：工具不是能力展示，而是行动的合法形式

一旦推理得出“需要行动”，系统就进入工具层。

这里最重要的一点是：工具不是模型能力的附属展示，而是模型在外部世界中行动的合法形式。

为什么要强调“合法形式”？

因为模型本身不会直接读文件、写代码、执行命令、操作网页。它只能通过被允许的工具接口间接行动。

这意味着工具在闭环里承担两个角色：

- 对外：连接真实世界。
- 对内：约束模型只能以某种受控方式行动。

所以一个 Agent 的工具系统，不只是“它能做多少事”，更是“它被允许怎样做事”。

## 3.6 反馈：没有反馈，就没有真正的闭环

很多系统看起来像 Agent，但本质上还只是“模型调用了一次工具然后结束”。这种结构最大的问题是：它没有形成真正的反馈回路。

反馈的作用是把动作结果重新送回系统，让下一轮推理不再建立在假设上，而建立在真实观察上。

例如：

- 文件已经读到了什么内容。
- 命令到底成功还是失败。
- 测试报了什么错。
- 浏览器页面现在处于什么状态。
- 上一步修改有没有产生副作用。

只有这些结果重新进入系统，后面的推理才是在“看见世界变化之后继续工作”。

这也是 ReAct 循环真正不同于静态问答的地方。它不是先想完再一次做完，而是每做一步，都重新观察世界，再决定下一步。

## 3.7 循环：循环不是重复，而是阶段性推进

闭环的“循环”也很容易被误解。它不是简单的多轮重复，更不是“让模型一直想下去”。

更准确的说法是：循环意味着系统可以基于反馈不断重新定位当前阶段。

一个成熟的循环通常会不断在下面几种状态之间切换：

- 信息还不够，需要继续查。
- 路径已经明确，可以继续执行。
- 当前尝试失败了，需要换方案。
- 局部任务已经完成，应该回到主目标。
- 风险开始升高，应该停下确认。

这意味着循环的关键不是“转了多少轮”，而是每一轮有没有把系统推向更接近完成、或者更接近明确终止的状态。

如果系统只是在重复相近动作、反复尝试同一条路、或者持续围绕局部问题打转，那不叫有效循环，那叫失控。

## 3.8 终止：不会停的系统，不叫成熟 Agent

很多看起来“很自主”的系统，最后真正暴露问题的地方，不在开始，而在停止。

终止条件不清楚，会直接带来三种风险：

- 系统明明已经完成了，还在继续做无意义动作。
- 系统明明应该停下确认，却默认继续推进。
- 系统遇到失败时，不知道该结束还是重试，于是开始打转。

因此，终止从来不是闭环最后随手补的东西，而是闭环一开始就必须被定义的部分。

一个成熟 Agent 至少要知道三类终止：

- 成功终止：任务已经达到完成条件。
- 人机交接终止：当前继续推进的风险或不确定性过高，需要交还给人。
- 失败终止：继续执行不会带来更高概率的成功，只会制造更多噪声或副作用。

如果系统不能区分这三种终止，它就很难真正稳定。

## 3.9 七个环节里，最容易被低估的是哪几个

从实践看，最容易被低估的通常不是推理本身，而是这三块：

### 上下文

很多团队高估模型推理，低估上下文组织，结果系统从一开始就工作在一个错误现场里。

### 反馈

没有真实反馈，系统只是在连续猜测，而不是连续行动。

### 终止

不会停的系统，最终一定会在长任务、权限风险或成本控制上出问题。

也正因为如此，闭环分析最有价值的地方，不是确认系统有没有“模型 + 工具”，而是确认它在这些容易被忽略的环节有没有真正做工程设计。

## 3.10 这把尺子后面怎么用

后面整本书对不同项目的分析，基本都会回到这七个问题：

- 目标是如何被明确的。
- 上下文怎样进入工作台。
- 推理在这里承担的到底是回答职责，还是控制流职责。
- 工具怎样被约束成合法行动面。
- 反馈结果是否被结构化回流。
- 循环怎样避免失控。
- 终止条件怎样定义。

这也是为什么第三章虽然看起来像基础章，实际上却是整本书最重要的入口之一。

一旦这把尺子建立起来，后面读任何一个 Agent 系统，都不会只是看它“酷不酷”，而会更自然地追问：

它的闭环到底是完整的，还是只是某一两步做得很亮眼。
