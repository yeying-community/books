# 第七章：Claude Code 的运行时骨架

理解 `Claude Code`，最容易走偏的方式是从工具列表开始。

工具当然重要，但如果一上来就数它有多少命令、多少工具、多少隐藏开关，很快就会迷失在局部。对工程师更有价值的切口，是先顺着一条运行时主链路看清楚：一条用户输入，究竟怎样穿过启动、Prompt 装配、主循环、工具执行和权限裁决，最后变成一个可持续运行的代理系统。

## 7.1 它首先不是“一次性 CLI”

官方概览对 `Claude Code` 的描述已经给出一个很关键的定位：它不是单纯的终端问答工具，而是一个能读代码库、改文件、跑命令、连接 IDE 和浏览器的 `agentic coding tool`。

这个定位决定了它的工程取向必须和普通 CLI 不同。

普通 CLI 关心的是命令执行一次是否成功；代码代理关心的是：

- 启动够不够快，能否频繁进入工作状态。
- 会话状态能否延续。
- 工具调用是否可控。
- 长任务是否能继续。
- 上下文过长时如何压缩而不失去主线。
- 高权限环境下如何把风险尽量前置。

很多社区文章都提到其启动阶段会并行预取一些信息。即使不把这些实现细节神化，这个观察仍然很有价值，因为它说明系统从一开始就在按运行时体验而不是按脚本调用来设计。

## 7.2 一条更值得看的主链路

把公开分析和现有文档放在一起看，`Claude Code` 至少可以被理解为这样一条主链：

1. 启动阶段恢复或预热必要状态。
2. 接收用户输入，准备会话级上下文。
3. 装配系统 Prompt、用户规则、系统环境和动态上下文。
4. 进入主循环，让模型基于当前状态决定下一步动作。
5. 通过工具层执行读、写、命令、搜索、外部连接等操作。
6. 在每个关键动作前后经过权限与模式裁决。
7. 当上下文过长时触发压缩；当任务变长时依赖记忆与状态续住。

这条链路比“它有哪些功能”更重要，因为系统的大部分工程判断都附着在这条链上。

## 7.3 Prompt 在这里是装配层，不是静态文案

很多人第一次看代码代理系统，会把 Prompt 理解成一大段“写得很厉害的系统提示词”。这在工程上是不够的。

从公开分析和 Anthropic 文档的描述看，更合理的理解是：

`Claude Code` 的 Prompt 更像一个运行时装配层。

它不是只把一段固定文案丢给模型，而是会把多类信息按角色和稳定性重新拼装：

- 静态系统规则。
- 用户长期规则，比如 `CLAUDE.md`。
- 当前项目环境。
- 当前会话状态。
- 当前可用工具与行为约束。

这件事的重要性在于，Agent 做得越深，Prompt 管理越不像写文案，越像在做编译和装配：哪些部分能复用，哪些部分必须动态计算，哪些部分属于稳定前缀，哪些部分必须随当前任务变化。

社区材料里提到工具顺序固定、工具自带行为规则、某些提示词被拆分成模块，这些都指向同一个工程事实：Prompt 已经成为运行时结构的一部分。

## 7.4 工具不是能力展示，而是契约系统

`Claude Code` 的另一个关键点，是它没有把工具当成“给模型更多自由”的渠道，而是把工具做成了行为契约。

这里至少有三层值得注意：

第一层，工具暴露了边界。模型不是直接拥有机器的一切能力，而是通过一组受约束的工具接口触达文件系统、终端、搜索和外部系统。

第二层，工具携带了行为规则。社区分析中反复提到“先读再改”“默认保守”“避免破坏性 git 命令”这类约束。这些约束是否完全覆盖所有场景另说，但它们说明 Anthropic 在有意识地把工程纪律前移到工具定义里。

第三层，工具执行结果会反馈回主循环。Agent 能否继续，不只取决于模型一次回答得多漂亮，而取决于工具输出是否足够结构化、是否能让下一轮决策继续推进。

因此，工具系统的价值不在“功能多”，而在它是不是一个可重复、可约束、可反馈的执行面。

## 7.5 权限系统不是一个弹窗，而是一条决策管道

很多人会把权限理解成“危险操作前弹一次确认”。这对代码代理来说太浅了。

在一个可以读写文件、执行命令、连接外部资源的本地 Agent 里，权限系统更像是一条连续裁决管道：

- 当前模式是否允许该动作。
- 该动作是否命中规则白名单或黑名单。
- 该动作是否需要提升确认级别。
- 该动作是否在项目配置、Hook、Skill 或 MCP 的影响范围内。

这意味着权限不是 UI 层的附属物，而是运行时的一部分。它直接影响工具是否可调用、命令是否可执行、长任务能否自动继续。

这也解释了为什么 `Claude Code` 这类系统在工程上值得研究，同时也必须谨慎：它的价值和风险来自同一件事，即模型开始具备了在本地环境中“确定性动手”的能力。

## 7.6 真正值得关注的不是“泄露了什么”，而是暴露了什么方法

围绕 `Claude Code` 的舆论很容易被“源码泄露”这件事本身吸走注意力。但站到工程学习的角度，更重要的是这次可见材料让社区更清楚地看到了几个方法论事实：

- 代码代理不是一个大 Prompt，而是一个运行时系统。
- Prompt、工具、权限、压缩、记忆必须一起设计。
- 高质量 Agent 的差异，往往出在上下文装配和执行约束，而不是单次回答。
- 长任务能力的前提不是“无限上下文”，而是有层次的状态管理。

后面章节会继续把这些点展开，尤其是最容易被混淆的两件事：记忆和上下文。
