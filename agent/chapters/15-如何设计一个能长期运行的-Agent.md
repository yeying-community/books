# 第十五章：如何设计一个能长期运行的 Agent

到了这一部分，重点就不该再停留在“看懂了哪些项目”，而应该开始收束成一套可以直接拿来设计系统的方法。

这套方法不追求抽象得漂亮，而追求一个更实际的目标：当你准备做一个 Agent 时，先想什么、后定什么、哪些边界必须提前画清楚，才能避免系统一开始就走向失控。

如果把前面理论篇和实践篇压缩成一句话，那就是：

一个能长期运行的 Agent，不是因为它一开始就很聪明，而是因为它在上下文、工具、状态、权限和终止条件上，都被设计成了可以持续工作。

下面按设计顺序展开。

## 15.1 第一步不是选模型，而是先定义任务边界

很多团队做 Agent 时，第一个问题就是“该用哪个模型”。这个问题当然重要，但它不应该排在最前面。

更靠前的问题应该是：

- 这个 Agent 到底替用户推进什么目标。
- 它最多自主推进到哪一步。
- 哪些动作必须停下来交还给人。
- 哪些结果算成功，哪些结果算失败。

如果这一步不清楚，后面所有工程决策都会发散。因为你根本不知道系统是在做：

- 单轮增强问答。
- 多步工具代理。
- 长任务执行助手。
- 半自动协作 Copilot。

这几类系统看起来都像 Agent，但需要的运行时完全不同。

因此，一个长期可运行的 Agent，第一条设计原则就是：先定义自主范围，再定义能力范围。

## 15.2 第二步是画最小工作闭环，而不是先列功能清单

系统能力越多，越容易让团队一开始就去列功能表：

- 支持哪些工具。
- 接哪些渠道。
- 有多少技能。
- 能不能多 Agent。

这些都重要，但它们都应该排在闭环之后。

真正应该先画的是最小工作闭环：

1. 目标从哪里进入系统。
2. 上下文如何被装配。
3. 模型依据什么做下一步判断。
4. 哪些工具可以被调用。
5. 工具结果如何反馈。
6. 系统如何继续下一轮。
7. 系统在什么条件下结束或请求确认。

没有这张闭环图，功能点越多，只会让系统越乱。

## 15.3 第三步是先分层，再加能力

很多 Agent 一开始失败，不是因为能力不够，而是因为几类信息和几类动作混在一起，没有分层。

至少有四层东西应该在设计早期就分开：

### 规则层

组织规则、项目规则、个人偏好、目录局部约束，不应该写在同一团 Prompt 里。

### 上下文层

长期规则、当前会话态、工具结果、临时读取材料，不能混成同一种上下文块。

### 状态层

工作记忆、短期任务状态、长期记忆，必须区分存放和回收方式。

### 执行层

读、写、命令、外部调用、浏览器操作，这些动作的风险完全不同，不能只因为都叫“工具”就放在一个粗糙接口里。

系统一开始分层越清楚，后面增加能力时越不容易引发结构性混乱。

如果希望这一步在设计评审里真正可用，可以继续追问四个问题：

- 规则层里，哪些规则是全局常量，哪些规则允许项目覆盖。
- 上下文层里，哪些信息要常驻，哪些信息必须按需加载。
- 状态层里，哪些状态只为当前任务服务，哪些状态允许跨任务复用。
- 执行层里，哪些动作默认开放，哪些动作必须显式申请。

这四个追问的价值在于，它们会逼着团队把“看起来都属于 Prompt 或工具的东西”重新拆回边界清楚的模块。

## 15.4 第四步是把工具当成边界，而不是当成卖点

很多团队喜欢把“我们支持多少工具”当作 Agent 的核心卖点。这个思路很容易把系统带偏。

对长期运行来说，工具最重要的作用不是展示能力，而是定义边界：

- 模型可以通过什么方式接触外部世界。
- 每类动作的输入输出长什么样。
- 哪些动作可以自动继续，哪些动作必须额外确认。
- 工具失败后，系统能否知道该重试、换路还是停止。

这意味着工具设计的优先级应该是：

1. 语义边界是否清晰。
2. 风险分级是否清晰。
3. 返回结果是否适合进入下一轮推理。
4. 再考虑工具数量和扩展性。

一个只有十个工具但边界清楚的系统，通常比一个有一百个工具但返回混乱的系统更可靠。

再往前走一步，工具契约至少应该包含四类信息：

- 工具是做什么的，不做什么。
- 输入参数的必填项、默认值和危险参数。
- 输出结果里哪些字段适合直接喂回模型，哪些字段只适合展示给人。
- 失败时返回的是可恢复错误、不可恢复错误，还是需要人工决策的错误。

如果没有这层契约，模型表面上是在“调用工具”，实际仍然是在和一团不稳定的执行面搏斗。

## 15.5 第五步是默认把权限系统放进核心架构

只要 Agent 能动手，权限就不是附属模块，而是核心模块。

这里最容易犯的错，是等功能差不多做完了，再在外面套一个确认框。这样做往往只解决了表层问题，没有解决真正的决策链问题。

权限系统至少要提前回答：

- 哪些动作属于低风险，可自动继续。
- 哪些动作属于高风险，必须单独治理。
- 项目配置、Hook、技能、脚本、外部服务会怎样放大执行面。
- 哪些操作是不可逆的，应该被视为特殊类别。

系统越强，权限越要前置。否则能力增长和风险增长会同步发生，而且后者常常比前者更快。

很多团队在这里还会漏掉一个问题：权限不只是“允不允许执行”，还是“以什么身份、在什么环境、带着什么上下文执行”。

这意味着权限设计至少要同时覆盖三层：

- 动作级权限：这次能不能执行。
- 环境级权限：这次在哪个沙箱、账号、目录、网络环境里执行。
- 身份级权限：这次代表谁执行，结果会落到谁的工作区和状态里。

只有把这三层合起来，权限系统才不至于沦为一个太晚才出现的确认按钮。

## 15.6 第六步是为失败设计，而不是只为成功路径设计

Agent 设计里一个很现实的原则是：不要默认系统会按理想路径运行。

真实系统会遇到：

- 工具失败。
- 数据冲突。
- 权限不足。
- 上下文过长。
- 模型误判。
- 环境状态变化。
- 用户中途打断。

因此，设计时必须提前考虑这些问题：

- 失败后是重试、换路还是停止。
- 当前状态如何保存。
- 中断之后如何恢复。
- 压缩之后如何不丢主线。
- 人类接手时能否快速看懂当前局面。

一个长期可运行的 Agent，不是永远不出错，而是出错后不会立刻散架。

从工程上说，这里最值得单独设计的是“恢复点”。

也就是每推进到一个关键阶段，系统是否能留下足够的信息，让下一轮还能继续：

- 当前目标是什么。
- 已经完成了什么。
- 哪些证据值得保留。
- 当前为什么停下。
- 下一步最合理的动作是什么。

如果没有恢复点，所谓的压缩、总结、checkpoint 往往只是把历史对话缩短了，并没有真的把任务状态保存下来。

## 15.7 第七步是把记忆当成状态系统的一部分，而不是神秘外挂

很多团队给 Agent 加“记忆”时，做法更像外挂：先把主循环搭起来，后面再找个向量库或数据库往上接。

这种做法最大的问题，是它经常没有回答三个基础问题：

- 为什么要记这条信息。
- 这条信息未来会在什么时机被取回。
- 这条信息过期后谁来清理。

如果这三个问题答不清，记忆系统很快就会从“增强能力”变成“污染上下文”。

更稳妥的设计方式是把记忆看成状态系统的延伸：

- 工作记忆服务当前主链路。
- 短期状态服务当前任务续航。
- 长期记忆服务跨任务复用。

这样一来，记忆就不再是一个额外加分项，而是任务状态如何跨轮、跨会话、跨周期保留下来的系统性设计。

## 15.8 第八步是从第一天起就准备评测面

很多 Agent 系统到了后期才发现，自己其实没有办法稳定判断“是不是变好了”。

所以评测面不能太晚才补。

至少要尽早定义几类指标：

- 工具调用正确率。
- 多步任务完成率。
- 中断恢复成功率。
- 权限拦截是否符合预期。
- 单任务平均成本和时延。

如果系统没有这些基本观测面，团队很容易被个别惊艳 demo 误导，误以为系统整体已经成熟。

如果再压实一点，评测面至少应该能区分三类失败：

- 模型判断失败。
- 工具执行失败。
- 运行时治理失败。

区分不开这三类失败，后续优化就很容易方向错误。

## 15.9 最后收束成一套设计顺序

如果把这一章压成一个简化版流程，可以这样理解：

1. 先定义任务目标和自主边界。
2. 再画最小工作闭环。
3. 然后把规则、上下文、状态、执行面分层。
4. 再设计工具契约和权限边界。
5. 同时设计失败恢复点和记忆分层。
6. 再把评测面和观测面接进来。
7. 最后才去扩展工具、入口、多 Agent 和生态。

这个顺序听起来保守，但它的价值非常直接：先让系统能持续工作，再让系统看起来强大。
