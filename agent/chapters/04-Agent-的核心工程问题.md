# 第四章：Agent 的核心工程问题

一旦接受 Agent 是一个运行时系统，而不是一个“会聊天的模型壳”，工程问题就会迅速浮现出来。

当前最核心的六类问题是：

## 4.1 上下文工程

系统到底该把哪些信息送进模型。

这包括规则分层、环境信息注入、目录级说明、对话历史裁剪、动态材料加载以及缓存稳定性。上下文工程做不好，Agent 很快就会要么失忆，要么发散。

## 4.2 工具系统

系统如何定义模型的行动面。

工具不是简单的 API 列表，而是执行边界、输入输出契约和结果回传机制。工具系统做不好，模型即使知道该做什么，也无法稳定把动作落地。

## 4.3 记忆系统

系统如何跨步骤、跨任务、跨会话保存有效信息。

上下文窗口不能替代记忆。成熟系统至少要区分工作记忆、短期状态和长期记忆。

## 4.4 权限与安全

系统如何限制高风险动作。

代码代理、浏览器代理和企业助手一旦能调用命令、脚本、外部服务，就会天然带来攻击面。权限系统必须进入架构核心，而不是停留在一个确认弹窗上。

## 4.5 恢复与长任务续航

系统在长流程中如何继续工作。

一旦任务跨越几十步，或者跨越多个会话，系统就必须处理压缩、续跑、状态恢复和中断接管问题。

## 4.6 评测与治理

系统如何知道自己到底有没有变好。

没有评测，Agent 系统很容易在“看起来更强”和“实际上更稳”之间混淆。工具调用正确率、长任务完成率、恢复成功率、错误面和成本，都是必须长期看的指标。

这六类问题，基本构成了后面分析所有项目时的第二把尺子。

## 4.7 这六类问题在实际框架里通常会继续细分

新增参考资料里有一组关于记忆、工具、执行循环、多 Agent 的系列文章。它们最大的价值，不在于给出最终结论，而在于提醒我们：这六类问题一旦进入真实框架，往往还会继续细化。

例如：

- 记忆问题会继续拆成“存什么、存在哪、怎么取、怎么管”。
- 工具问题会继续拆成“能力暴露、权限检查、执行拦截、结果注入”。
- 执行循环会继续拆成“规划、执行、错误恢复、停止条件、状态续航”。
- 多 Agent 会继续拆成“角色分离、上下文隔离、通信方式、结果汇总”。

这些拆法很有用，因为它们能帮助我们避免一种常见误区：把复杂问题用一个大词糊过去。

比如你说“记忆系统很重要”，这没错，但如果不继续问“到底存什么、怎么取、谁来更新”，这个判断就还停留在口号层。

同样地，你说“多 Agent 很强”也没错，但如果不继续问“它们怎么隔离、怎么通信、怎么避免协调成本吃掉收益”，这个判断也不够工程化。

因此，从这六类核心问题往下再细分，是后面进入项目对照时非常必要的一步。它能让我们分析的不是抽象能力，而是具体设计决策。
