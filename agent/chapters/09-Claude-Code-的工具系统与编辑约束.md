# 第九章：Claude Code 的工具系统与编辑约束

如果说 Prompt 装配解决的是“模型在什么现场里做判断”，那么工具系统解决的就是“模型到底能以什么方式动手”。

这是 `Claude Code` 和普通聊天产品最本质的差别之一。

聊天产品的价值主要来自回答；代码代理的价值来自回答之后还能继续执行：读文件、搜目录、改代码、跑命令、看结果、再继续下一轮。模型一旦进入这个链路，工具系统就不再是附属能力，而是整个运行时的执行面。

## 9.1 工具不是“超能力”，而是受约束的机器接口

很多人第一次看 Agent，会把工具理解成给模型加上一堆外挂：能查文件、能跑命令、能联网、能调用外部服务，于是模型似乎突然“有手了”。

这个理解只对了一半。

工具更准确的角色，是把原本无限的机器能力收敛成一组有限、明确、可回传结果的接口。模型不是直接操作整个操作系统，而是通过这些接口间接行动。

这背后的工程意义很大：

- 行动面被收束了，系统才可能审计。
- 每类动作有清晰输入输出，主循环才可能继续推进。
- 权限系统才能针对工具种类、参数形态和执行风险做裁决。

也就是说，工具不是“让模型无所不能”，而是“让模型在可描述、可判断、可阻断的接口上行动”。

## 9.2 为什么读、写、执行必须是三种不同问题

在代码代理里，最常见的三个动作看起来很像：

- 读文件。
- 改文件。
- 执行命令。

但从工程上看，这其实是三种风险截然不同的问题。

### 读文件

读文件主要解决信息获取问题。它的风险相对较低，但仍然决定了后续判断质量。如果模型读得太少，它就会臆测；如果读得太多，它就会把无关噪声带进上下文。

### 改文件

改文件是最核心、也最需要约束的动作。因为代码代理一旦进入写操作，系统就必须处理两个问题：

- 模型是不是基于真实文件内容在改，而不是凭记忆补全。
- 改动是否足够局部、可解释、可恢复。

### 执行命令

执行命令的破坏面最大，因为它可以改变文件、环境、依赖甚至远程状态。一个看似普通的命令，可能在项目里触发脚本、构建链、安装流程、git 动作或外部网络请求。

把这三类动作拆开，是任何成熟代码代理的起点。只要系统在语义上把它们混成一件事，后面的权限、安全和恢复都很难做好。

## 9.3 “先读再改”不是风格建议，而是核心纪律

社区对 `Claude Code` 的一个普遍观察，是它非常强调类似“先读再改”的行为规则。这个点值得保留，而且需要明确写成工程纪律，而不是写成习惯建议。

原因很简单。

代码代理最危险的行为之一，就是在没有读取当前文件真实状态的前提下直接生成补丁。这样做会带来三个直接后果：

- 容易覆盖用户刚改过但尚未提交的内容。
- 容易依据过时上下文做出错误编辑。
- 容易把局部问题扩展成大面积重写。

因此，“先读再改”真正保护的不是代码美观，而是事实来源。模型必须尽量先接触真实文件，再决定修改范围和修改方式。只有这样，写操作才不是基于幻觉的二次创作。

这也是为什么一个好的代码代理通常会偏好：

- 先搜索和定位。
- 再读取足够上下文。
- 然后进行小范围、目标明确的编辑。
- 最后通过命令或测试验证结果。

它看起来更慢，但这是用更多步骤换更小的错误面。

## 9.4 Git 安全规则为什么重要

社区文章里反复提到一类规则：避免破坏性 `git` 命令，除非用户明确要求。

这类规则表面上只是“别乱用 git”，实际上它说明 `Claude Code` 把版本控制系统也视为高风险执行面，而不是普通工具。

原因在于，对代码代理来说，`git reset --hard`、`checkout --`、大范围回滚、强推、自动提交这类操作的风险，不只是技术性的，还带有很强的协作破坏性：

- 可能清掉用户未提交的工作。
- 可能覆盖本地实验状态。
- 可能把代理自己的错误扩大成仓库级事故。

所以，把 `git safety` 写进工具行为规则，本质上是在做一件非常现实的事：承认代理没有资格默认替用户做不可逆决策。

这个原则未来不只适用于 `git`。任何能够批量、不可逆地改变系统状态的接口，都应该被单独视为高风险面。

## 9.5 编辑约束真正服务的是可恢复性

很多人谈编辑约束时，只会从“提高准确率”这个角度理解。但对代码代理来说，约束还有一个同样关键的目标：可恢复性。

一个可恢复的编辑系统，通常具备几个特征：

- 修改范围尽量局部。
- 修改前尽量基于真实内容定位。
- 修改后容易通过 diff 或测试解释。
- 出问题时容易回退到较小范围。

这类设计的价值，在长任务里尤其明显。因为长任务不是一次改完就结束，而是会反复经历：

- 读代码。
- 做假设。
- 改动。
- 运行命令。
- 根据新结果再改。

如果每次改动都过大、过散、过依赖模型记忆，系统很快就会进入不可恢复状态。用户和代理都说不清哪一步把事情搞坏了。

因此，编辑约束不是保守主义，而是把失败成本控制在一个人类还能接手的范围内。

## 9.6 工具系统的真正门槛不在数量，而在结果形态

很多项目喜欢用“支持多少工具”做展示，但对运行时而言，真正难的不是把工具数量堆上去，而是保证工具结果能稳定进入下一轮决策。

一个工具系统想真正可用，至少要回答三个问题：

- 返回结果是否足够结构化，能让模型知道发生了什么。
- 错误是否足够明确，能让模型决定重试、换路还是停止。
- 输出是否被适当裁剪，避免一次调用把上下文污染掉。

这也是为什么成熟 Agent 的工具层，往往同时在做两件事：

- 执行机器动作。
- 重新包装机器结果，使其可被后续推理消费。

从这个角度看，工具层并不只是 adapter，它同时也是推理前处理和结果后处理的一部分。

## 9.7 技能不是长 Prompt，而是可分发的能力单元

围绕 Anthropic Agent Skills 的一些官方最佳实践和社区总结，给工具系统补上了另一个重要视角：当能力开始被封装成 `Skill` 时，它就不再只是“多写一段 Prompt”，而是一个可发现、可加载、可执行、可治理的能力单元。

这类材料很有价值，但使用时要注意边界。具体字段限制、产品行为和 YAML 语义应该以官方文档为准；更值得吸收进本书的，是它背后的工程模式。

第一，技能需要被正确发现。

如果系统只能在会话开始时看到技能的名称和描述，那么 `description` 就不是普通摘要，而是路由信号。它应该说明这个技能做什么、什么时候用、哪些关键词或场景应该触发。同时，成熟的技能还需要写清楚“不该什么时候用”。否则技能一多，就会出现多个技能抢同一类任务，或者没有技能愿意接某个边界任务。

第二，技能要服从上下文经济。

一个常见误区，是把所有说明、API 参考、示例、边界情况都塞进 `SKILL.md`。这样做看起来完整，实际上是在用每一次对话支付全量上下文成本。更合理的形态，是把主文件写成入口和索引，把长参考、领域资料和示例拆到按需加载的文件里。脚本实现也不应该默认进入上下文；如果某段逻辑稳定、可重复、能被测试，就更适合放进 `scripts/` 里，由模型调用并读取结果。

第三，技能指令要按风险调节自由度。

开放型任务可以给模型判断空间；结构化输出可以用模板和示例约束形态；高风险批量操作则应该先生成计划，再验证计划，最后才执行真实动作。这个差别很关键，因为“把指令写得更死”并不总是更安全。过窄的指令会让系统在边界情况里做不了事，过宽的指令又会让系统在高风险动作上越界。技能设计真正要控制的，是每一步允许多少偏差。

第四，技能要显式记录已知陷阱。

很多技能最有价值的内容，不是正常流程，而是“这里以前踩过什么坑”。库版本差异、平台差异、空结果、字段缺失、命令副作用，这些内容如果不写进技能，模型就会在真实环境里重新试错。已知陷阱本质上是一种经验记忆，但它必须保持更新，否则旧坑会变成新的误导。

第五，可验证流程要优先于一次性生成。

对代码、配置、结构化文档和批量修改来说，技能最好内置“生成、验证、修复、再验证”的循环。对于不可逆或大范围操作，更稳的做法是先产生可检查计划，把验证放在副作用之前。这样一来，技能不是让模型更大胆，而是把模型的自由度收束在可检查的中间产物上。

这些原则说明，技能系统的重点不是“给模型更多提示词”，而是把一类可复用能力做成一套小型运行时契约：什么时候触发、加载多少上下文、能调用哪些工具、如何验证结果、失败后如何收束。

## 9.8 从 Claude Code 可以沉淀出的工具设计原则

把现有材料合在一起看，`Claude Code` 至少帮助社区更清楚地看到几条工具设计原则：

1. 工具首先是边界，不是能力炫耀。
2. 读、写、执行必须分开治理。
3. 写操作必须尽量建立在真实读取之上。
4. 高风险命令不能默认自动化。
5. 编辑约束的目标不只是准确率，还有可恢复性与协作友好性。
6. 工具结果必须适合进入下一轮推理，而不是把原始机器输出原封不动塞回上下文。
7. 技能要被当成可治理能力单元，而不是一段可以无限增长的长 Prompt。

等下一章继续拆权限管道时，这些原则会进一步连到 Hook、MCP、技能和项目级配置上。到那时就会更清楚地看到：为什么一个代码代理既强大，又天然带有攻击面。
