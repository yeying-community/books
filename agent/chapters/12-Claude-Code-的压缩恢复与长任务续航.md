# 第十二章：Claude Code 的压缩、恢复与长任务续航

如果说前面几章解决的是 `Claude Code` 怎样开始工作，那么这一章要解决的，就是它怎样在工作很多步之后仍然不散架。

这是代码代理和普通聊天系统最容易被低估的分界线之一。

一个普通问答系统只需要处理当前回合；一个真正有价值的代码代理，必须面对更难的问题：

- 上下文会不断膨胀。
- 中间状态会持续累积。
- 工具输出会把工作台越堆越满。
- 用户会打断、切换方向、要求继续。
- 系统会在几十步之后逐渐失去重点。

所以，长任务能力从来不是“上下文窗口够大”这么简单，而是一整套压缩、恢复和状态续航机制共同作用的结果。

## 12.1 为什么长任务天然会失真

长任务的第一个敌人，不是模型不够聪明，而是任务执行过程中信息会不断失真。

这种失真主要来自四个方向：

第一，历史越来越长。

每一步搜索、读取、编辑、测试、报错都会往上下文里继续堆材料。即使窗口暂时还装得下，模型也不可能同样敏锐地关注每一条历史。

第二，局部材料会滞留。

很多信息只在某一阶段有用，比如某次 grep 结果、一次失败的构建日志、一个临时文件路径。但如果这些内容长期留在工作台上，它们就会逐渐污染后续判断。

第三，目标会漂移。

任务跑到后半段之后，模型很容易开始围绕最近的局部问题打转，而忘记最初的任务目标和完成标准。

第四，中断是真实存在的。

用户会切走、会追问别的问题、会要求系统稍后继续。没有恢复机制的代理，一旦断开主线，就只能重新开始。

这四类问题放在一起，决定了长任务系统一定要处理“怎么让系统忘得合理、记得关键、断了还能接上”。

## 12.2 压缩的本质不是省 token，而是重建工作台

很多人一提到压缩，就会立刻想到成本优化或者上下文溢出保护。

这些都对，但都还不够。

对代码代理来说，压缩更准确的角色，是重建工作台。

前面已经讲过，上下文更像工作台而不是档案室。工作台一旦堆满，问题不只是贵，而是乱。乱到一定程度之后，系统即使还“记得很多东西”，也已经很难继续抓住真正重要的线索。

因此，压缩要做的，不是机械地删 token，而是重新组织当前局面：

- 当前任务到底已经推进到哪里。
- 哪些历史步骤只需要保留结论。
- 哪些失败尝试应该折叠成经验，而不是原样保留。
- 哪些局部日志应该从工作台挪走。
- 哪些尚未解决的问题必须继续留在前景。

一套好的压缩机制，结果看起来不像“被删掉很多信息”，而像“系统重新整理了一次桌面”。

## 12.3 记忆、压缩、恢复是三件相关但不同的事

这里最容易混淆，所以必须拆开。

### 记忆

解决的是哪些信息值得跨步骤、跨任务、跨会话保存。

### 压缩

解决的是当前工作台太乱时，哪些内容该保留原文，哪些内容该变成摘要，哪些内容该离开当前上下文。

### 恢复

解决的是系统在中断之后，如何重新拿回足够的状态，让任务继续，而不是重新开一个相似但已经断线的新会话。

三者当然互相关联，但不能混成一种机制。

如果把压缩当记忆，系统会把大量局部摘要当成长期知识，最后越积越脏。

如果把记忆当恢复，系统会以为长期 learnings 足够支撑继续执行，但实际上缺的恰恰是这次任务的中间状态。

如果没有恢复机制，压缩做得再好，也只能帮助当前会话，不足以支撑长跨度续航。

## 12.4 Claude Code 值得借鉴的地方，不在“有压缩”，而在“承认必须压缩”

社区分析里提到 `Claude Code` 存在多级压缩与 compact 流程，这类具体实现细节后续仍可继续深挖。但就当前可见材料来说，更值得先提炼出来的，是它明确承认了一件事：

长任务里，压缩不是异常路径，而是正常路径。

这一点非常重要。

很多系统一开始只把压缩当成“超限之后的紧急补丁”，结果会导致两个问题：

- 平时完全不为压缩后的可继续性做设计。
- 一旦进入压缩，系统就像被迫重启了一次，只留下很粗的总结。

而成熟一点的系统会把压缩视为长任务运行时的一部分，提前为它设计：

- 哪些状态必须被明确保留。
- 哪些工具结果可以被折叠。
- 哪些待办和阻塞必须高亮。
- 哪些上下文边界在压缩后仍应保留。

这件事本质上不是算法细节，而是运行时态度。

## 12.5 恢复能力的核心，不是“记得很多”，而是“知道从哪继续”

恢复机制最容易被误写成一种宽泛的“记忆能力”。这并不准确。

对长任务来说，恢复真正需要的是一种更具体的状态表达：

- 当前任务目标是什么。
- 已经做过哪些关键动作。
- 哪些尝试失败过，以及为什么失败。
- 当前阻塞点是什么。
- 下一步最应该做什么。

这五类信息，才决定了系统能不能“从这里继续”，而不是重新绕一圈。

这也是为什么一个恢复友好的代理，往往更强调阶段性结论，而不是原始流水账。

流水账适合审计，不适合恢复。

恢复需要的是可执行状态，而不是全文回放。

## 12.6 长任务续航最终依赖的是状态表达质量

很多团队会把长任务问题归因于模型窗口、模型推理或模型价格。但从工程上看，更根本的变量往往是状态表达质量。

如果系统无法清楚表达：

- 当前在做什么。
- 为什么这样做。
- 已经排除了哪些路径。
- 现在卡在哪。
- 接下来应该优先做什么。

那么无论窗口多大，系统迟早都会发散。

相反，如果状态表达做得足够好，即使发生压缩、切换、暂停和恢复，系统也更容易重新抓回主线。

这也是为什么“任务摘要”不应该只是写给人看的汇报，而应该被当成运行时的一种中间结构。它既服务于模型，也服务于人类接手。

## 12.7 对社区最有用的长任务设计原则

把 `Claude Code` 当前暴露出来的思路和前面的理论合在一起，可以先沉淀出几条很实用的原则：

1. 长任务一定会膨胀，压缩要被视为正常运行机制。
2. 压缩的目标不是删 token，而是重建工作台秩序。
3. 记忆、压缩、恢复必须分开设计。
4. 恢复依赖的是可执行状态表达，而不是全文历史。
5. 任务越长，越要把“当前目标、已完成事项、阻塞点、下一步”显式化。
6. 系统越早承认会中断，就越容易真的支持续航。

## 12.8 为什么这一章会影响后面的方法论

长任务能力不是锦上添花，而是会反过来影响整个 Agent 设计。

因为一旦你认真考虑压缩、恢复和续航，前面的很多设计都会被重新约束：

- Prompt 不能只是堆材料，必须能支持阶段切换。
- 工具结果不能只追求完整，必须考虑后续可折叠性。
- 权限系统不能只管当前一步，还要考虑长链执行风险。
- 记忆系统不能只追求积累，还要区分任务态和长期态。

也就是说，长任务问题不是实践里的一个附属章节，而是会倒逼整个运行时结构更加成熟。

这也是为什么在 `Claude Code` 之后再看 `OpenClaw` 与 `Harness`，很多问题会更容易进入同一套分析框架。
