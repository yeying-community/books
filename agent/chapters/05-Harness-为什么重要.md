# 第五章：Harness 为什么重要

很多团队在做 Agent 时，一开始最关注的是模型：换更强的模型、更长的上下文、更好的推理能力。

这些当然重要，但如果只盯着模型，很快就会遇到一个现实：同样的模型，换一个系统壳子，效果可能完全不同。

这就是 `Harness` 这个概念真正重要的地方。

`Harness` 可以粗略理解为：把模型能力稳定接出来的那套运行时系统。它不直接提供推理能力，但决定了推理能力能否在真实任务中持续发挥作用。

为什么说它重要？

因为 Agent 的很多失败，并不是模型不会，而是系统没把事情组织好：

- 上下文太乱，模型找不到重点。
- 工具暴露太粗，模型不知道如何可靠调用。
- 没有短期状态，跑到第十步像回到第一步。
- 权限边界太松，系统一强就危险。
- 没有恢复机制，长任务一断就从头开始。

从这个角度看，模型决定能力上限，Harness 决定稳定性下限。

这也是为什么后面看 `Claude Code`、`OpenClaw`、`Harness` 本身时，重点不该只是“它用了什么模型”，而应该是“它如何组织上下文、工具、记忆、权限、恢复和评测”。

一旦读者接受这一点，很多对 Agent 的争论都会变得更清楚：真正值得研究的，不只是模型智能，而是运行时怎样让智能变成可重复的系统行为。

## 5.1 为什么很多“模型问题”其实是系统问题

在 Agent 讨论里，一个很常见的坏习惯，是把几乎所有失败都统称为“模型不够稳定”。

这种说法有时没错，但多数时候过于偷懒。因为模型只是运行时里最显眼的一层，不是唯一的一层。

比如下面几种现象：

- 模型总是抓不住重点。
- 模型经常调错工具。
- 模型跑到十几步之后开始发散。
- 模型在恢复任务时像失忆了一样。
- 模型一进入项目环境就做出风险很高的动作。

这些现象当然可能和模型能力有关，但同样可能是：

- 上下文组织方式有问题。
- 工具描述不清楚。
- 状态没有被正确表达。
- 权限边界没有进入主循环。
- 压缩之后丢掉了关键任务信息。

这也是为什么 `Harness` 很重要。它迫使我们把“模型失败”继续往下拆，直到能落到更具体的系统职责上。

换句话说，`Harness` 的价值之一，不是让 Agent 立刻更强，而是让系统失败时更可诊断。

## 5.2 Harness 本质上在替模型承担什么

如果把问题再说得更直接一些，可以这样理解：

模型本身擅长的是在给定上下文里做模式匹配、推理和生成；而 `Harness` 负责的是把这个过程放进真实世界里还能持续运转。

它至少替模型承担了六类责任。

### 第一，组织上下文

模型没有能力自己决定“哪些信息现在应该进上下文，哪些应该暂时不进”。如果没有运行时做这件事，系统通常会走向两个极端：

- 要么信息太少，模型只能瞎猜。
- 要么信息太多，模型被噪声淹没。

### 第二，暴露工具

模型也不会天然知道自己能以什么方式安全地接触外部世界。工具的边界、参数、返回结果、错误语义，都需要由运行时定义。

### 第三，保存状态

模型本身不具备跨步骤、跨任务的持久状态。短期状态如何保存，长期记忆如何沉淀，中断之后如何恢复，都不是模型自己能解决的问题。

### 第四，限制权限

模型不会自然理解“这个动作虽然能做，但不应该默认做”。哪些动作要自动继续，哪些动作要停止确认，哪些动作是高风险类别，这些都必须由运行时治理。

### 第五，处理失败

真实系统里，工具会报错、环境会变化、脚本会失败、数据会冲突。失败后是重试、换路、停止还是交还给人，这些也都属于运行时职责。

### 第六，评估表现

模型不会自己告诉你“我这次为什么成功、那次为什么失败”。评测、观测、回归验证这些能力，也必须由系统层提供。

当这六类责任被清楚地看见之后，`Harness` 这个概念就不再抽象了。它不是一个行业流行词，而是模型和真实系统之间那层真正让事情跑起来的结构。

## 5.3 为什么没有 Harness，Agent 很容易停在 Demo 阶段

很多 Agent Demo 一开始都很迷人，因为它们往往在一个非常受控的场景里展示了几个连续动作：

- 理解问题。
- 调用工具。
- 返回结果。

这在几分钟的演示里已经足够让人觉得系统“像个人一样工作”。

但一旦进入真实环境，问题就会迅速暴露：

- 用户输入不再规整。
- 工具数量开始增加。
- 任务路径不再固定。
- 错误和异常变成常态。
- 需要跨会话继续。
- 需要进入组织安全边界。

如果没有 `Harness`，系统就会暴露出非常典型的 Demo 症状：

- 一次成功很惊艳，但无法稳定复现。
- 简单任务有效，长任务很快失控。
- 功能越加越多，但整体表现反而更乱。
- 一旦环境稍微变化，系统就像换了个脑子。

这也是为什么很多团队后来会发现，真正难的不是“让模型调用一个工具”，而是“让系统持续几十步仍然像在做同一件事”。

## 5.4 Harness 和基础模型，不是替代关系

这里也需要避免另一个误区：既然 `Harness` 这么重要，是不是模型就不重要了？

当然不是。

一个差的运行时，确实会浪费好模型；

但一个很强的运行时，也不能把非常差的模型变成真正可用的 Agent。

更合理的理解是：

- 基础模型决定能力上限。
- Harness 决定这些能力在真实任务里能保留多少。

模型越强，运行时越有机会把复杂任务做深；

运行时越成熟，模型越不容易把能力浪费在噪声、错误边界和断裂状态上。

所以，讨论 Agent 不能滑向任何一个极端：

- 不能把所有功劳都归给模型。
- 也不能假装模型不重要，仿佛只靠系统工程就能替代推理能力。

真正成熟的分析方式，是把两者分开看，再看它们怎样耦合。

## 5.5 从 Harness 视角回看前面所有问题

如果用 `Harness` 的视角回看本书前面几章，会发现很多分散的问题其实都开始自然归位：

- `Agent` 为什么会出现：因为静态交互不够，系统需要把模型接进运行过程。
- `Agent` 和 `Workflow` 为什么不同：因为一部分控制流从开发期迁移到了运行时。
- 最小工作闭环为什么重要：因为 Harness 要把目标、上下文、工具、反馈、循环和终止串起来。
- 核心工程问题为什么是上下文、工具、记忆、权限、恢复和评测：因为这些本来就是运行时的核心职责。
- 成本为什么不只是 `token`：因为运行时在承担复杂度转移带来的额外治理成本。

从这个角度讲，`Harness` 不是本书里的一个局部主题，而更像是一条隐含主线。它帮助我们把“Agent 工程”从零散技巧，重新看成一套系统设计问题。

## 5.6 对社区最有用的结论

如果把这一章再压成几个最值得带走的结论，可以保留下面这些：

1. Agent 的稳定性主要来自运行时，不只来自模型。
2. 很多被归因于“模型不稳定”的问题，实际属于上下文、工具、状态或权限设计问题。
3. Harness 的价值不仅是让系统更强，更是让系统更可诊断、更可治理。
4. 一个没有 Harness 的 Agent，通常只能短暂地像 Demo 一样工作。
5. 判断一个 Agent 是否成熟，不能只问它用了什么模型，还要问它的运行时到底承担了哪些职责。

有了这层框架，后面读 `Claude Code`、`OpenClaw`、`Harness` 本身时，就不会只停留在“它有哪些功能”，而会开始真正看清：它到底把哪部分复杂度接住了，又把哪部分复杂度留给了模型，或者留给了用户。
