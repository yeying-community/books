# 第十一章：Claude Code 的权限管道与攻击面

如果前面几章主要解决的是 `Claude Code` 为什么值得研究，那么这一章要解决的，就是它为什么必须被谨慎研究。

原因很简单。

一个只能回答问题的模型，风险相对有限；一个能读文件、改代码、跑命令、接入外部系统、接受项目配置影响的本地代码代理，风险结构就完全不同了。

它的价值和风险来自同一件事：模型开始具备了在真实环境里持续动手的能力。

因此，这一章的重点不是制造惊悚感，而是把风险传播路径讲清楚。只有这样，社区才不会一边高估 Agent 的神秘能力，一边低估它作为执行系统的攻击面。

## 11.1 权限系统为什么不是一个确认弹窗

很多人第一次接触代码代理时，会自然把权限理解成一种交互动作：

- 危险命令弹个确认。
- 写文件前给个授权。
- 跑脚本前让用户点一次“继续”。

这当然是权限的一部分，但远远不够。

对 `Claude Code` 这类系统来说，权限更准确的角色是一条运行时决策管道。它至少要持续参与下面几件事：

- 当前模式是否允许该类动作。
- 当前工具是否允许在这一轮被调用。
- 某个命令是否属于高风险命令。
- 某个项目配置是否扩大了执行面。
- 某个结果是否足以支撑继续自动推进。

这意味着权限不是 UI 层的附属物，而是运行时控制链的一部分。它决定的不是“用户愿不愿意点确定”，而是系统有没有资格继续推进下一步动作。

从架构上看，这一层越早进入设计，系统后面越不容易出现“能力已经做出来了，但根本不知道该如何收边”的问题。

## 11.2 为什么本地代码代理天然带攻击面

社区里经常会出现一种误解：觉得本地运行就天然更安全。

这其实过于乐观。

本地运行确实减少了一部分远端托管风险，但也把系统直接放进了用户真实开发环境：

- 它能看到工作目录。
- 它可能读到仓库里的配置文件。
- 它能执行项目脚本。
- 它可能接触本地凭证、环境变量和工具链。
- 它可能被仓库中的说明文件、技能定义、Hook 或外部扩展间接影响。

换句话说，本地并不等于低风险。本地意味着系统和真实环境的接触更直接，风险传播链更短。

这也是为什么 `Claude Code` 这种系统的安全问题，不该只被理解成“模型会不会胡说”，而应该被理解成“模型在什么边界里会触发确定性执行”。

## 11.3 攻击面一：项目级规则与上下文投毒

前面已经讲过，`Claude Code` 会读取 `CLAUDE.md` 这类规则文件，并支持递归发现与导入机制。

从可用性角度看，这非常合理，因为它让项目规则可以自然进入上下文；但从风险角度看，它也意味着项目本身可以影响代理的行为方式。

这里的风险不一定表现为显眼的恶意脚本，更常见的是更隐蔽的上下文投毒：

- 在项目规则里加入误导性行为要求。
- 用“团队规范”包装高风险动作建议。
- 通过导入链把用户不易察觉的局部规则带入系统。
- 让模型把不可信记忆当作高优先级行动依据。

这类风险的要害在于：它不一定直接执行危险操作，但会先改变模型的判断前提。一旦判断前提被污染，后续工具调用就可能在“看起来合理”的情况下逐步滑向高风险路径。

因此，项目规则系统带来的风险，不只是“会不会读错文档”，而是“谁在定义代理默认相信什么”。

## 11.4 攻击面二：命令执行与脚本链式触发

命令执行是本地代码代理最强的能力之一，也是最危险的能力之一。

危险并不只来自显式的破坏命令。更常见的情况是链式触发：

- 一个看似普通的测试命令，可能触发项目脚本。
- 一个安装命令，可能执行 `postinstall`。
- 一个构建流程，可能调用外部服务、生成文件、覆盖缓存。
- 一个开发命令，可能进一步触发 shell 脚本或 Python、Node、Makefile 里的动作。

也就是说，代理并不是在执行“一个孤立命令”，而是在启动一条可能继续扩散的执行链。

这就是为什么前面讲工具系统时强调：执行命令不能和读文件、改文件视为同一种动作。它的破坏面大得多，而且往往不完全暴露在第一层命令文本里。

从风险治理角度看，一个成熟的权限系统至少要承认这件事：

- 命令是否高风险，不能只看命令名。
- 是否允许自动继续，不能只看当前一步。
- 项目脚本本身也是执行面的一部分。

## 11.5 攻击面三：Hooks、Skills、MCP 扩大了系统边界

这也是当前社区最容易被轻描淡写的部分。

只要系统支持 `Hooks`、`Skills`、`MCP` 或类似扩展机制，它的边界就不再只是“模型 + 内建工具”，而是开始进入“模型 + 可编排执行插件面”。

这类机制的价值当然很大：

- `Hooks` 让某些动作前后可以插入固定逻辑。
- `Skills` 让能力可以被模块化扩展。
- `MCP` 让外部工具和服务更容易接入。

但价值和风险是同步增长的。

因为每一种扩展机制都在做同一件事：让系统能够在模型外再接一个新的确定性执行入口。

从攻击面角度看，这意味着：

- 行为边界不再只由核心产品定义。
- 项目或组织可能通过配置改变代理执行路径。
- 外部工具定义和能力描述会影响模型判断。
- 用户很难仅靠主界面意识到系统当前真实拥有的执行面。
- 技能里的工具预批准、脚本目录和参考文件，会把“能力描述”和“执行能力”绑到一起。

这也是为什么不能只把扩展生态当成“能力繁荣”，而必须同时把它看成“边界变复杂”。

尤其是 `Skills`。一个技能表面上只是任务说明，但如果它同时声明了可用工具、携带了脚本、引导模型按某个流程执行，那么它实际上已经进入运行时边界。技能的 `description` 会影响路由，参考文件会影响判断前提，脚本会影响确定性执行，工具配置会影响自动化程度。

因此，技能治理至少要追问三件事：

- 这个技能是否只拿到了完成任务所需的最小工具集合。
- 技能里的脚本和参考资料是否来自可信来源，是否会被项目内容间接覆盖。
- 技能声明的工具可用性是预批准，还是已经有更硬的权限策略兜底。

这能帮助我们避免一个常见误判：以为技能只是提示词资产，风险只在模型是否听话。实际上，一旦技能能调工具、跑脚本、影响计划，它就应该被纳入权限和供应链治理。

## 11.6 什么叫配置投毒

“配置投毒”这个词现在很容易被讲得过于戏剧化，但如果换成更工程化的表述，它其实描述的是一类很朴素的风险：

系统不是直接被人攻击，而是被带有误导性、扩权性或恶意副作用的配置间接改变了行为。

在 `Claude Code` 这类系统里，比较典型的传播路径可能是：

1. 仓库或依赖中存在会被代理读取的规则、说明、脚本或扩展定义。
2. 这些内容改变了模型对“当前应该做什么”的判断。
3. 模型调用了原本受信任的工具接口。
4. 工具又触发了更深一层的命令、脚本或外部连接。

注意，这条链路里每一步单独看都可能“像是正常功能”：

- 读规则是正常功能。
- 参考项目说明是正常功能。
- 调工具是正常功能。
- 跑测试或构建也是正常功能。

真正的问题在于，这几步串起来之后，系统可能在用户没有完全意识到的情况下，沿着一条被污染的路径执行下去。

因此，配置投毒最需要警惕的，不是它够不够“黑客电影”，而是它足够像正常开发流程。

## 11.7 Claude Code 值得借鉴的防线思路

虽然公开材料无法证明 `Claude Code` 已经完美解决了这些问题，但至少能看出一些值得借鉴的防线方向：

### 默认保守

对高风险命令、破坏性 `git` 动作和大范围修改保持保守倾向，不把“自动做更多事”当成第一优先级。

### 显式授权优先

把某些动作明确视为需要额外确认的类别，而不是让模型在模糊边界里自己猜。

### 把不可逆动作单独治理

可逆错误和不可逆错误不是一回事。前者通常可以通过重新编辑、重试或局部回退恢复；后者则可能直接造成协作或环境事故。

### 工具边界先于模型自由度

模型再聪明，也不应该优先于工具边界和权限规则。换句话说，系统宁可少做一点，也不能先把执行面完全打开，再指望模型自己克制。

### 让用户能接手

一套真正有用的安全设计，不只是拦住坏事，还要在系统停下来时让人类能迅速看懂当前状态、决定是否继续、以及从哪里接手。

## 11.8 对社区最重要的判断

围绕 `Claude Code` 的安全讨论，最重要的不是证明它“危险”还是“安全”，而是把一个更成熟的判断建立起来：

只要一个 Agent 开始进入真实执行环境，它就必须被当成运行时系统来治理，而不是被当成聊天界面的自然延伸。

这会直接影响社区后续怎么看待：

- 项目规则文件。
- 自动命令执行。
- 技能市场和插件生态。
- Hook 和外部协议接入。
- 本地代理与组织治理的关系。

只有在这个前提下，后面再继续写 `OpenClaw`、`Harness` 或其他系统时，安全和权限才不会永远沦为最后才补的一页“注意事项”。
