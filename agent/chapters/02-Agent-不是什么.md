# 第二章：Agent 不是什么

讨论 Agent 时，最容易出问题的地方，不是定义太少，而是大家把很多不同东西都叫成了 Agent。

为了避免后面分析失焦，先把几个常见概念切开。

## 2.1 Agent 不是普通聊天机器人

聊天机器人主要解决的是对话生成问题。它的核心价值在于理解问题、组织语言、生成回答。

Agent 则更进一步。它不仅要回答，还要在回答之后继续行动：读外部信息、调用工具、观察结果、修正路径，并在必要时连续跑多轮。

因此，两者最大的差别不在“有没有对话界面”，而在“有没有执行闭环”。

## 2.2 Agent 不是 Copilot 的同义词

Copilot 更强调人在主导，模型在旁边补全、建议、提醒、辅助。

Agent 则更强调系统在一个明确目标下自主推进若干步骤，即使中间仍然需要人类确认。换句话说，Copilot 更像副驾，Agent 更像被限制在边界内的执行代理。

## 2.3 Agent 也不是 RAG 的升级版

RAG 主要解决的是“模型不知道外部知识”的问题，也就是先检索，再注入。

Agent 虽然常常包含 RAG，但它处理的不只是知识获取，还包括：

- 选择哪些工具。
- 决定何时重试。
- 判断是否需要追问。
- 在多步执行里维护状态。
- 在失败后决定换路还是停止。

因此，RAG 是 Agent 可能使用的能力之一，而不是 Agent 的完整定义。

## 2.4 Agent 也不等于 Workflow

这是最值得仔细讲清楚的一点。

严格说，很多 Agent 系统最终也会表现出一种工作流形态：有输入、有步骤、有工具、有条件分支、有结束条件。真正的区别在于控制流在哪一侧。

`Workflow` 的控制流主要由工程师在开发期显式写死：

- 什么时候调用哪个工具。
- 哪些条件进入哪条分支。
- 哪些错误如何重试。
- 哪些异常如何降级。

`Agent` 则把其中一部分控制流迁移到了运行时，让模型在上下文和工具边界内决定：

- 先做什么。
- 失败后要不要重试。
- 该不该换工具。
- 是否还缺关键上下文。

这也是为什么 Agent 往往更贵、更慢，但在长尾和复杂环境里更灵活。

## 2.5 因此，Agent 最准确的理解方式

更准确的说法是：

Agent 不是某个具体产品形态，而是一类系统设计范式。

它的核心不是“更像人”，而是把问题从静态请求，变成带有目标、上下文、工具、反馈和循环的动态执行过程。

有了这个边界，后面再看任何项目，就不会因为它接了聊天入口、能调函数、或者能做多轮对话，就轻易把它误判成真正的 Agent。

## 2.6 为什么“先排除错误对象”很重要

很多技术书一上来喜欢先下定义，但对 Agent 这个话题，先排除错误对象往往更有用。

原因很简单：Agent 不是一个天然边界清楚的术语。它太容易被借来包装各种不同东西：

- 带工具调用的聊天机器人被叫 Agent。
- 带流程编排的平台被叫 Agent。
- 带 RAG 的问答系统被叫 Agent。
- 能改代码的 Copilot 被叫 Agent。
- 甚至只要支持多轮对话，也常被叫 Agent。

如果这一步不先切开，后面几乎所有讨论都会漂移。你会发现不同人嘴里都在说“Agent”，但实际上说的是完全不同的系统。

这会直接带来两个坏结果：

- 架构判断失真：明明是 `workflow` 问题，却拿 Agent 的方式解决。
- 评估标准错位：明明该看恢复和权限，却只盯着回答质量。

所以这一章的真正任务不是给 Agent 一个漂亮定义，而是帮读者建立“哪些对象不该先被归进来”的边界感。

## 2.7 聊天机器人为什么通常还不算 Agent

很多产品已经很会聊天，也能记住对话历史，甚至能接一点外部搜索。这些能力会让它们看起来和 Agent 很接近。

但从系统结构上看，普通聊天机器人通常仍然缺少两个关键点：

- 它没有稳定的执行闭环。
- 它不以持续推进任务为目标。

聊天机器人最核心的职责仍然是“理解并回答当前问题”。即使它支持多轮上下文，多数时候也只是让回答更连贯，而不是让系统进入一个多步骤运行过程。

这意味着聊天机器人更像：

- 对话接口。
- 知识访问入口。
- 语言交互层。

而不是一个真正意义上的运行时代理。

只有当系统开始明确承担“持续推进目标、调用工具、接收反馈、继续执行、在必要时结束或交还控制权”的职责时，它才开始越过聊天机器人的边界。

## 2.8 Copilot 为什么也不能直接等同于 Agent

`Copilot` 是另一个非常容易混淆的对象，尤其在代码场景里。

很多代码助手已经能：

- 读当前文件。
- 生成修改建议。
- 提供命令。
- 帮你解释报错。

这些能力当然很接近 Agent，但 `Copilot` 的默认位置仍然是辅助位，而不是代理位。

这里的区别不在于“会不会用工具”，而在于控制权默认落在哪一侧。

`Copilot` 更像：

- 人在主导。
- 模型在旁建议。
- 决策和执行确认主要由人承担。

`Agent` 更像：

- 系统在明确目标下推进若干步骤。
- 人更多承担边界设定、授权和关键节点干预。
- 模型和运行时共同承担一部分控制流。

这两者没有高低之分，只是分工不同。

很多真正可用的系统，最后也会同时包含这两种模式：

- 有些时候用户需要一个 Copilot，随问随答、随看随改。
- 有些时候用户需要一个 Agent，让系统把一串事情连着做下去。

把这两者混为一谈，往往会导致预期错位。用户以为自己得到的是“可持续推进的代理”，实际拿到的却只是“更主动一点的助手”。

## 2.9 RAG 为什么只解决了 Agent 的一部分问题

`RAG` 对 Agent 非常重要，但它解决的只是 Agent 的一个子问题：外部知识怎么进入模型。

这件事当然关键，因为模型不知道最新世界，也不知道公司内部知识库、代码仓库、工单系统和浏览器页面里发生了什么。

但即使你把 RAG 做得很好，系统也仍然不自动变成 Agent。因为 Agent 还必须解决：

- 什么时候检索。
- 检索之后做什么。
- 是否需要继续调用别的工具。
- 当前信息够不够支撑下一步。
- 是继续推进，还是停下来确认。

也就是说，RAG 主要在解决“模型该知道什么”，而 Agent 还在解决“系统接下来该怎么行动”。

这两层问题不能混。

一个系统可以是优秀的 RAG 系统，但不是 Agent；

一个 Agent 也可以用到 RAG，但它的复杂度远远不止检索这一层。

## 2.10 Workflow 为什么是最容易被误判的邻居

真正和 Agent 最像、也最容易被误判的，其实不是聊天机器人，也不是 RAG，而是 `Workflow`。

因为两者表面上都像是在“把事情一步步做下去”：

- 都可能调用工具。
- 都可能有条件分支。
- 都可能处理异常。
- 都可能需要多轮步骤。

所以，单看外观，很多 Agent 和 Workflow 甚至会长得很像。

真正的分界线仍然是那句最重要的话：

控制流主要写在开发期，还是主要落在运行时。

`Workflow` 更像：

- 工程师提前写好路径。
- 什么时候分支、什么时候重试、什么时候结束，都由显式逻辑描述。

`Agent` 更像：

- 工程师定义目标、边界、工具和运行时约束。
- 具体某一步怎么走、是否需要换路、是否需要继续补信息，由模型在运行时判断。

这也是为什么一个系统即使支持很多工具、多轮执行和错误处理，也未必就已经是 Agent。只要它的主控制流仍然基本靠显式逻辑写死，它就更接近 Workflow。

## 2.11 多 Agent 也不自动等于更成熟的 Agent

还有一个常见误区是：只要系统支持多 Agent，就会显得更高级，仿佛天然跨进了下一代架构。

这也不成立。

多 Agent 只是复杂度管理的一种手段，不是成熟度证明。

如果一个系统：

- 单 Agent 闭环都不完整。
- 状态表达不清楚。
- 工具边界混乱。
- 权限系统没进核心。

那么再加几个子 Agent，通常只会把问题放大，而不是解决问题。

从架构上看，多 Agent 应该解决的是：

- 上下文隔离。
- 子任务并行。
- 角色分工。
- 结果汇总。

如果它没有解决这些问题，只是“起了更多实例一起跑”，那就还停留在功能表象，不足以说明系统已经进入更成熟阶段。

## 2.12 最后把边界收束成一句话

如果一定要把这一章压缩成一个最值得保留的判断，可以写成：

Agent 不等于会聊天，不等于会检索，不等于会调工具，也不等于有多轮对话。

Agent 真正成立的前提是：系统已经从一次性回答器，变成了一个围绕目标持续组织上下文、调用工具、接收反馈、控制循环并决定终止的运行过程。

只有跨过这条线，后面的记忆、权限、恢复、执行循环、多 Agent 才会真正成为架构问题，而不是功能包装。
