# 第六章：Agent 的成本本质

讨论 Agent 时，很多人只盯着一个数字：`token`。

这当然重要，但不够。

更完整的理解是：Agent 往往是在用更多运行时成本，换更低的显式编排成本，以及更强的长尾任务适应能力。

这里的运行时成本至少包括：

- 更多轮模型推理。
- 更多次工具调用。
- 更高的延迟。
- 更高的失败重试概率。
- 更复杂的状态管理。

如果继续往下拆，这些成本还可以再分成五类。

## 6.1 模型成本只是第一层

最容易看见的，是模型调用本身的账单。

Agent 相比普通问答系统，通常会天然更贵，因为它经常不是一轮完成，而是多轮推进：

- 先理解任务。
- 再判断要不要调工具。
- 再根据工具结果继续推理。
- 再尝试修正、补充、确认和总结。

这意味着同一个用户请求，在 Agent 里往往会变成一串连续推理，而不是一次生成。

如果再叠加更长的上下文、更复杂的工具描述、更频繁的压缩和恢复，模型成本自然会上升。

但如果只看到这一层，就很容易误判。因为很多团队最后撑不住的，并不是账单本身，而是后面几层看起来没那么显眼的成本。

## 6.2 第二层成本是时延

很多 Agent 在 demo 里看起来很强，但一到真实环境，用户最先感受到的往往不是“贵”，而是“慢”。

慢的来源通常不只一个：

- 模型要多轮推理。
- 工具调用本身有网络和执行时延。
- 工具失败后还要重试。
- 中间可能要读文件、压缩、恢复、切换阶段。

换句话说，Agent 的输出不是单次生成延迟，而是一整条链路延迟。

这对场景选择影响很大。

如果用户需要的是秒级稳定响应，比如高频表单、批量处理、标准报表，这类场景对时延非常敏感，Agent 的优势往往很难体现。

但如果任务本身就需要几分钟甚至更久，而且关键价值在于能不能把事情真正做完，那么更长的时延就未必是决定性问题。

因此，判断 Agent 是否值得上，不只是看每次调用多少钱，还要看用户是否真的能接受这种链式延迟。

## 6.3 第三层成本是失败率和波动性

这是很多团队最容易低估的一层。

传统 `workflow` 虽然不一定聪明，但路径稳定，失败模式也比较明确。Agent 则不同，它把一部分控制流迁移到了运行时，于是系统天然会带来更高的波动性：

- 这次选对了工具，下次可能就换错了。
- 这次补槽很自然，下次可能漏掉关键问题。
- 这次压缩后还能继续，下次压缩后可能丢主线。
- 这次命令顺利执行，下次可能卡在脚本副作用。

这意味着 Agent 的真实成本，不只是平均成本，还包括波动成本。

波动成本会带来两种非常现实的后果：

- 你必须投入更多评测和治理，去确认系统不是“偶尔很强，整体很飘”。
- 团队和用户会花更多精力处理“为什么这次和上次不一样”。

所以在生产视角里，一个更贵但稳定的系统，常常比一个平均成本看似更低、但波动很大的系统更容易长期存活。

## 6.4 第四层成本是维护成本

这也是“用 token 换架构”这个框架真正成立的地方。

Agent 之所以值得做，不是因为它更便宜，而是因为它有机会降低另一种更难受的成本：显式编排维护成本。

当环境复杂度继续增加时，传统显式工作流会开始出现分支爆炸：

- 新接一个系统，就多一层适配。
- 多一种异常，就多一条分支。
- 多一类输入组合，就多一个决策条件。
- 多一次失败重试策略，就多一段显式逻辑。

这些东西一开始看起来都能补，但随着系统越来越大，维护成本会变得非常难看。

Agent 的意义就在这里：它不是把复杂度消灭了，而是把其中一部分从开发期显式逻辑，转移到了运行时判断。

这就解释了为什么很多团队愿意接受更高的模型成本和更高的时延，因为他们在换另一种更昂贵的东西：

- 更少的分支逻辑。
- 更快接入新工具。
- 更容易覆盖长尾问题。
- 更少针对每种异常逐条硬编码。

当然，这个交换只有在长尾复杂度足够高时才成立。否则你只是用更复杂的系统去处理本来就能写死的流程。

## 6.5 第五层成本是组织成本

这是最不容易被技术讨论覆盖，却最影响落地的一层。

一旦系统从 `workflow` 变成 Agent，团队内部的协作方式也会跟着变化：

- 产品不能再只定义页面和按钮，还要定义自主边界。
- 工程不能只写 happy path，还要设计恢复与权限。
- 测试不能只测固定输入输出，还要测多轮行为和异常路径。
- 运维和安全不能只盯部署，还要看执行面和扩展面。

这意味着 Agent 的引入，常常不只是技术替换，而是组织协作模式升级。

如果团队没有准备好面对这种变化，Agent 很容易变成一个“谁都能提需求，但没人能对整体行为负责”的系统。

所以，组织成本也是选型的一部分。不是所有团队都适合在当前阶段上 Agent，也不是所有问题都值得团队为 Agent 改变协作结构。

而它换来的通常是：

- 少写一部分硬编码分支。
- 更容易接入新工具和新环境。
- 更容易覆盖不可穷举的异常与长尾需求。
- 在复杂任务里保留更强的动态调整能力。

因此，Agent 的问题从来不是“贵不贵”这么简单，而是“这笔成本换来的灵活性，是否真的值得”。

## 6.6 什么叫“值得”

这里最需要避免的是泛泛而谈。所谓“值得”，至少可以拆成四个判断问题：

1. 当前场景的复杂度，是否真的主要来自不可穷举性。
2. 运行时灵活性，是否真的能替代掉大量显式编排工作。
3. 增加的模型成本、时延和波动，是否能被任务价值覆盖。
4. 团队是否有能力治理由此带来的权限、恢复、评测和运维问题。

只要这四个问题里有两三个答案是否定的，那么 Agent 往往就不值得上得太重。

反过来，如果一个场景具备下面这些特征，Agent 的投入通常更有意义：

- 任务目标复杂但价值高。
- 异常路径和输入组合很多。
- 工具和环境变化频繁。
- 人工显式编排已经开始失控。
- 组织愿意投入额外治理成本。

也就是说，Agent 最适合的不是“任何智能任务”，而是“复杂度已经高到值得用运行时灵活性去换”的任务。

这里也就能解释一个常见误区：

为什么很多简单任务其实不应该用 Agent。

如果一个任务路径稳定、异常有限、输入输出清楚，那么显式 `workflow` 往往更快、更便宜、更可控。只有当任务组合、外部环境和异常路径开始显著不可穷举时，Agent 的运行时灵活性才会体现出价值。

所以，判断一个场景该不该上 Agent，不能只看“模型能不能做”，而要一起看：

- 场景复杂度。
- 长尾比例。
- 工具和环境变化频率。
- 显式编排维护成本。
- 运行时成本是否能被业务价值覆盖。

这会成为后面第三部分“什么时候该用 Agent，什么时候该用 Workflow”的基础。

## 6.7 最后把成本问题收束成一句话

对 Agent 来说，真正昂贵的从来不只是 `token`，而是你选择把复杂度放在哪一侧。

你可以把复杂度放在开发期，写成越来越厚的显式流程、分支和异常处理；

也可以把其中一部分放到运行时，让模型和工具系统去动态承担。

前者通常更稳定、更可控，但扩展和维护会越来越重；

后者通常更灵活、更适合长尾，但治理和波动成本会明显上升。

因此，Agent 的成本本质，不是“模型贵”，而是“你愿不愿意用更高的运行时成本，去交换更低的显式编排复杂度，以及更强的长尾适应力”。
