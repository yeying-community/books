# 第十六章：什么时候该用 Agent，什么时候该用 Workflow

这是整本书里最现实的一章。

因为很多团队真正的问题，不是“Agent 怎么做”，而是“这个场景到底该不该上 Agent”。如果这个判断一开始就错了，后面无论上下文工程、工具设计、权限系统做得多认真，系统仍然会在错误方向上消耗大量时间和成本。

所以这一章不讨论概念热度，只讨论一个工程问题：什么场景应该用 Agent，什么场景应该老老实实用 Workflow。

## 16.1 先给结论：不是越复杂的系统就越高级

很多人容易把 Agent 当成 Workflow 的升级版，仿佛只要用了模型驱动控制流，系统就自动更先进。

这是错的。

Agent 和 Workflow 不是先进与落后的关系，而是两种不同的系统组织方式。它们各自适合的问题不同，代价结构也不同。

`Workflow` 的优势在于：

- 路径确定。
- 行为可预期。
- 成本稳定。
- 调试直接。
- 合规和审计更容易做。

`Agent` 的优势在于：

- 对长尾任务更灵活。
- 面对环境变化更容易调整。
- 对不可穷举异常更有适应性。
- 能把一部分显式编排迁移到运行时。

因此，真正的问题不是“谁更强”，而是“你的问题更像哪一种”。

## 16.2 最适合用 Workflow 的场景

如果一个任务具备下面这些特征，通常优先考虑 `Workflow`：

### 路径稳定

任务步骤基本固定，先做什么、后做什么、哪些条件进入哪条分支，大致都能提前写清楚。

例如：

- 固定字段抽取。
- 固定报表生成。
- 标准审批流转。
- 固定数据同步。
- 明确的表单填充和系统录入。

### 输入输出清楚

系统输入格式稳定，输出格式要求也稳定，中间不需要太多临场判断。

### 异常相对有限

错误情况虽然存在，但种类不多，而且能被显式列出来处理。

### 风险较高

一旦动作出错，代价很高，比如涉及财务、权限变更、正式发布、批量写库、客户外发。这类场景通常更需要可控和可审计，而不是运行时灵活性。

### 对成本和时延敏感

如果任务量大、频率高、响应时间要求严格，Agent 的多轮推理和工具调用成本往往不划算。

一句话概括：能提前穷举的流程，优先交给 Workflow。

## 16.3 最适合用 Agent 的场景

如果一个任务具备下面这些特征，Agent 的价值通常会更明显：

### 任务组合不可穷举

用户问题不是固定模板，而是会不断组合变化。

例如：

- 先解释问题，再对比多种方案，再去查资料，再生成结构化结论。
- 先分析代码，再找相关文件，再尝试修复，再跑测试，再继续修改。
- 先查多个外部系统，再根据结果选择下一步工具。

### 环境变化频繁

工具、数据源、网页结构、文件布局、知识来源经常变化，显式编排维护成本会越来越高。

### 异常路径太多

权限不足、字段缺失、接口限流、上下文不够、结果冲突、脚本失败，这些异常并不是少数，而是常态。

### 任务需要阶段性判断

系统不能只照流程跑，还需要不断判断：

- 现在信息够不够。
- 要不要补一个问题。
- 应该换工具还是继续当前路径。
- 当前结果值不值得进入下一步。

### 任务价值高于运行成本

如果一次成功完成任务带来的价值足够高，那么多花一些 `token`、多走几轮推理、增加一些工具调用延迟，是可以接受的。

一句话概括：当问题的复杂度主要来自不可穷举性时，Agent 才真正划算。

## 16.4 不要把“接了工具”误判成 Agent 适配场景

一个常见误区是：只要任务需要调用工具，就觉得应该上 Agent。

这也不对。

调用工具本身不是判断依据。关键在于工具调用是固定编排，还是动态决策。

例如：

- “读取 A 表，清洗后写入 B 表，再发通知”通常更像 Workflow。
- “根据用户问题动态决定查哪个系统、组合哪些信息、是否继续追问”才更像 Agent。

工具只是执行面。真正区分 Workflow 和 Agent 的，是控制流究竟在开发期，还是在运行时。

## 16.5 一个实用判断法：看复杂度主要落在哪

如果你需要在项目早期快速判断，可以用一个很实用的问题：

这个系统的复杂度，主要落在流程编排，还是落在运行时判断？

如果复杂度主要是：

- 固定流程多。
- 系统连接多。
- 字段映射多。
- 审批规则多。

那么优先考虑 Workflow。

如果复杂度主要是：

- 用户意图组合多。
- 异常路径多。
- 工具选择多变。
- 现场判断很多。

那么 Agent 更可能值得做。

这个判断法不华丽，但非常实用，因为它直接对应开发成本会落在哪一侧。

如果希望把这个判断法变成团队内部能复用的评审方法，可以直接做一个四维判断表：

- 路径稳定度高，偏向 `Workflow`。
- 运行时判断密度高，偏向 `Agent`。
- 错误代价高，偏向 `Workflow` 或 `Workflow` 包 `Agent`。
- 单次任务价值高且长尾变化多，偏向 `Agent`。

这四维里，只要前三维都明显偏左，通常就不值得强行 Agent 化。只有当“运行时判断密度”和“长尾价值”都明显上升时，Agent 的成本结构才开始合理。

换句话说，Agent 不是为“复杂流程”准备的，而是为“复杂判断”准备的。

## 16.6 很多成熟系统最终是“Workflow 包着 Agent”

现实里，最常见的最佳解不是二选一，而是组合。

很多成熟系统最终会长成下面这种形态：

- 大框架是 `Workflow`。
- 某些高不确定性节点内部嵌一个 `Agent`。

这种组合非常常见，也很合理。

因为很多业务系统的主路径其实是稳定的，真正难的是其中少数几个局部节点：

- 复杂问题理解。
- 动态资料检索。
- 多工具组合判断。
- 长尾异常处理。
- 代码修复和调试。

这些点适合交给 Agent；但外层的审批、状态流转、审计、重试策略、人工确认，往往仍然更适合 Workflow。

这类结构的好处是：

- 主流程仍然可控。
- 高不确定性部分获得灵活性。
- 成本不会无限外溢。
- 安全边界更容易画清楚。

因此，对很多企业系统来说，真正现实的答案不是“全 Agent 化”，而是“把 Agent 用在最该用的节点上”。

这里可以把混合架构再说得更具体一点。

一个比较稳妥的落地方式通常是：

1. 外层 `Workflow` 负责接入、鉴权、审计、预算、重试和人工确认。
2. 中间把真正高不确定性的节点切出来，例如复杂理解、开放检索、代码修复、异常归因。
3. 内层 `Agent` 只在这些节点里拥有有限自主性。
4. `Agent` 的产物再回到 `Workflow`，由后者决定是否继续推进正式动作。

这种结构的关键，不是“Workflow 调了一个 Agent API”，而是控制权仍然握在可审计主链路里。

也正因为如此，很多团队最开始就应该把问题改写成：

- 哪些节点必须保持确定性。
- 哪些节点值得让模型临场判断。
- 哪些节点即使模型判断错了，也有外层兜底。

这样做，比一开始讨论“要不要全量上 Agent”要成熟得多。

## 16.7 不该上 Agent 的几个危险信号

如果出现下面这些信号，通常说明你可能正在错误地使用 Agent：

### 只是为了显得先进

如果团队上 Agent 的主要原因是“现在大家都在做”，而不是业务上真的存在不可穷举复杂度，最后大概率只会得到一个更贵、更慢、更难调试的系统。

### 问题本质上是流程工程

如果真正麻烦的是系统对接、字段治理、权限流程、数据质量，而不是运行时决策，那么上 Agent 只会掩盖根问题。

### 失败代价极高但边界没定义清楚

如果系统一旦出错就可能造成生产事故、财务风险或合规问题，但团队又没有清晰的权限边界、确认机制和审计面，这类场景不适合贸然 Agent 化。

### 没有评测能力

如果团队根本没法判断 Agent 到底是在变好还是变坏，系统很快就会陷入“靠体感做决策”的状态。

### 把不确定性误判成智能需求

有些项目看起来很乱，但乱的原因并不是需要模型决策，而是上游数据质量差、流程所有权不清、接口治理混乱。

这类问题就算接入 Agent，也只是把原来的混乱包装成新的运行时复杂度。模型不会替团队补齐缺失的系统治理。

### 希望用 Agent 掩盖产品定义不清

如果团队连“系统到底替谁解决什么问题”都还没说清楚，就急着讨论多 Agent、记忆、技能生态，通常说明产品边界本身还没成立。

在这种情况下，Agent 往往会成为推迟产品收敛的一种借口。

## 16.8 三个常见反例：看起来像 Agent，其实不一定需要

为了避免概念误判，再看三个很常见的反例。

### 反例一：固定报销审核

表面上它要读材料、查规则、给结论，好像很适合 Agent。

但如果实际规则稳定、输入结构固定、异常类型可列举，那么更合理的做法通常是：

- 用 `Workflow` 跑主流程。
- 在个别非结构化票据理解节点上嵌模型能力。

也就是说，这类场景更像“Workflow 加局部模型能力”，而不是“完整 Agent”。

### 反例二：批量网页信息采集

如果目标站点稳定、采集字段固定、失败重试规则明确，这仍然更像流程自动化问题，而不是 Agent 问题。

只有当网页结构变化频繁、采集目标需要临场判断、还要动态切换工具和策略时，Agent 的价值才会明显出来。

### 反例三：标准客服问答

如果问题类型有限、答案模板稳定、升级路径明确，这类系统更适合检索加规则流转。

只有当客服场景开始出现复杂任务编排、跨系统查询、长链路判断和持续跟进时，才值得认真考虑 Agent。

## 16.9 最后给出一套选型顺序

如果把这一章收束成一套实操顺序，可以这样做：

1. 先问任务路径能否被提前穷举。
2. 再问复杂度主要来自流程，还是来自运行时判断。
3. 再问失败代价是否允许系统保留一定自主性。
4. 再问运行时成本是否被任务价值覆盖。
5. 如果答案偏稳定、偏可控、偏高风险，就先做 Workflow。
6. 如果答案偏长尾、偏多变、偏不可穷举，再考虑引入 Agent。
7. 如果两边都占一部分，就用 Workflow 包 Agent。

这套顺序的价值，在于它能让团队把“该不该用 Agent”从情绪判断，拉回工程判断。

如果还想再压缩成一个更硬的评审问题，其实只要问五句：

1. 这个问题的主复杂度，究竟是流程复杂，还是判断复杂。
2. 这部分判断，能不能被规则继续显式化。
3. 即使判断错了，系统有没有外层兜底。
4. 这部分灵活性，值不值得支付更高的成本和治理复杂度。
5. 如果今天不用 Agent，系统是否真的无法推进。

这五句问完，很多“看起来该做 Agent”的项目，其实会自然回到更朴素也更稳的工程方案上。

到这里，第三部分其实已经收住了。

前面一整本书做的事情，可以归纳成一句话：

Agent 不是为了替代所有系统，而是为了处理那些已经开始超出显式编排舒适区的问题。

真正成熟的团队，不是看到什么都想 Agent 化，而是知道该在哪些地方引入 Agent，又该在哪些地方坚持用更朴素的系统方法。
