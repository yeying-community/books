# 第十章：记忆与上下文，不是一个东西

很多 Agent 讨论一旦进入“上下文窗口越来越大”这个话题，就会很自然地滑向一个误解：上下文越长，记忆问题就越轻。

这并不成立。

无论是 `Claude Code` 的社区分析，还是 `Harness` 方向的工程总结，都反复指向同一个事实：上下文窗口不是记忆系统，最多只是记忆系统的一部分，而且是最昂贵、最脆弱、最容易变乱的一部分。

## 10.1 上下文窗口更像工作台

把上下文理解成工作台，比把它理解成档案库更接近工程现实。

工作台有三个特点：

- 面积有限，堆太多材料就会混乱。
- 适合处理当前任务，不适合保存全部历史。
- 工作结束后会被清空，天然不具备持久性。

这也是为什么只靠“把历史全塞进上下文”很快就会遇到三个问题：

- 容量问题：长任务下，工具返回、搜索结果、编辑差异、日志输出会迅速膨胀。
- 质量问题：即使窗口还没满，噪声上升也会让模型更难抓住关键点。
- 持久性问题：跨会话、跨任务、跨工具时，上下文天然保不住。

因此，上下文管理的核心不是“尽量多塞”，而是“当前必须保留什么”。

## 10.2 Claude Code 里的记忆边界

Anthropic 官方文档已经明确给出两种记忆来源：

- `CLAUDE.md`：用户明确写给系统的持久化规则。
- 自动记忆：系统在项目工作过程中积累的 learnings。

这已经足够说明一件事：`Claude Code` 自己也没有把记忆等同于上下文窗口，而是把“用户显式规则”和“系统运行中沉淀出的经验”分开处理。

社区分析对这套机制做了进一步拆分，常见提法包括：

- 项目级、个人级、组织级规则文件。
- 项目目录下的自动记忆索引与主题文件。
- 定期整理、压缩或重写记忆的后台过程。

这些细节里，有些需要继续核验，有些已经可以作为合理推断采纳。但无论实现细节怎样变化，设计边界已经很清楚：记忆是独立于单轮上下文的持久层。

## 10.3 记忆系统至少应该分层

`Harness` 方向最值得借鉴的地方之一，是它把记忆问题拆回三个层次：

### 工作记忆

当前步骤立刻需要的信息。通常直接体现在上下文窗口里，包括当前目标、最近动作、关键工具结果和临时约束。

### 短期记忆

属于本次任务周期，但不必一直挂在上下文里。它更像任务状态存储，保存“做过什么、发现了什么、下一步卡在哪里”。

### 长期记忆

跨任务、跨会话仍有价值的信息，比如用户偏好、项目架构约束、常见故障经验、历史决策结论。

这三层不是抽象概念游戏，而是对工程代价的真实划分：

- 工作记忆最快，但最贵。
- 短期记忆更稳，但访问要经过回收和召回。
- 长期记忆最持久，但必须接受筛选、更新和遗忘机制。

## 10.4 记忆系统的难点不在“存”，而在“写、读、更新、忘”

社区文章里最容易被低估的，是记忆系统后半段的工程复杂度。

真正的难点不只是把信息存下来，而是四个问题：

### 写什么

不是所有信息都值得写进记忆。原始日志、完整工具返回、一次性的中间输出通常价值很低。真正值得保存的，是会影响后续决策的事实、约束、偏好和经验。

### 什么时候读

每一步都全量检索记忆，成本高且噪声大；完全不读，等于没有记忆。比较合理的做法通常是事件驱动，比如新任务开始、进入新阶段、遇到阻塞、需要恢复中断状态时再召回。

### 如何更新

记忆会过时，会冲突，会被修正。系统必须接受“新信息推翻旧信息”这件事，而不是把所有旧内容永久堆积。

### 如何遗忘

如果没有遗忘机制，记忆系统最终会从知识库退化成噪声库。时间衰减、版本替换、冲突检测和定期整理，都属于这个问题的一部分。

## 10.5 对社区最有用的结论

围绕 `Claude Code`、`OpenClaw`、`Harness` 看一圈之后，关于记忆和上下文，至少可以先记住五个结论：

1. 更大的上下文窗口不能替代记忆系统。
2. 记忆系统必须分层，否则长任务几乎一定失控。
3. `CLAUDE.md` 这类规则文件解决的是显式约束，不解决隐式经验积累。
4. 自动记忆如果没有筛选、整理和遗忘机制，很快会反过来污染系统。
5. 真正成熟的 Agent，不是“记住所有东西”，而是知道什么该放工作台，什么该进档案柜，什么时候再取回来。

这也是后续我们继续拆工具路由、压缩和恢复机制时的基础前提。
