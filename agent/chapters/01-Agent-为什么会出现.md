# 第一章：Agent 为什么会出现

Agent 的出现，不是因为大家突然不满足于聊天机器人，而是因为传统模型交互方式在真实任务里很快会撞上边界。

这些边界至少有四类。

第一，模型不知道最新世界。

模型的知识主要来自训练期数据，而现实世界是持续变化的。新闻、代码仓库、公司内部文档、工单系统、监控数据、知识库和浏览器页面都在不断变化。只靠训练权重，模型不可能天然知道这些变化。

第二，模型不能天然动手。

一个普通聊天模型即使能给出方案，也不等于它能去读文件、改代码、操作终端、点击网页、调用公司内部系统。很多实际价值，就卡在“知道怎么做”和“真的去做”之间。

第三，真实任务不是单轮问答。

企业和开发场景里的很多任务都具有多步性：先搜集信息，再做判断，再调用工具，再根据结果修正路径。这个过程天然需要循环、反馈、重试和阶段切换。

第四，复杂环境不可穷举。

工具会变、接口会变、权限会失败、数据会冲突、网页会改版、脚本会报错。人可以为常见流程写 `workflow`，但当异常和组合开始爆炸时，显式编排成本会迅速变得难看。

这就是 Agent 真正填补的空白。

Agent 解决的不是“模型怎么回答得更好”，而是“如何把问题编译成一个可执行、可反馈、可继续的运行过程”。它把一部分原本由工程师在开发期写死的控制流，迁移到运行时由模型和工具系统共同承担。

因此，理解 Agent 的关键，不是把它神化成自治生命体，也不是把它贬低成单纯的脚本套壳，而是承认它是一种新的系统组织方式：用更多运行时推理和工具调用成本，换更低的显式编排复杂度，以及更强的长尾适应能力。
