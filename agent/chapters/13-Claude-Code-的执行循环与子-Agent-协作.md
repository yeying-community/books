# 第十三章：Claude Code 的执行循环与子 Agent 协作

如果说前面的几章分别解决了 `Claude Code` 的上下文、工具、权限和长任务问题，那么这一章要解决的，是这些模块在运行时里到底怎样被串起来。

这也是一个代码代理最接近“心脏”的部分。

因为无论记忆系统多漂亮、工具系统多完整、权限边界多清楚，如果执行循环本身设计得很差，系统最后仍然会出现一批非常典型的问题：

- 接到复杂任务，直接冲进去执行，中途才发现方向错了。
- 工具失败之后，不是无限重试，就是直接僵住。
- 长任务里每一步都在局部修补，但整体目标越来越模糊。
- 子任务明明可以拆开，却仍然挤在一个上下文里互相污染。

所以，执行循环不是“模型加一层 while loop”这么简单，而是 Agent Runtime 真正开始显露工程深度的地方。

## 13.1 执行循环为什么是 Agent 的心脏

在最抽象的层面，现代 Agent 几乎都可以归结为一个 `ReAct` 式循环：

1. 接收输入。
2. 根据当前状态推理下一步。
3. 调用工具执行动作。
4. 观察结果。
5. 把结果重新注入系统，继续下一轮。

这套结构听起来不复杂，但一旦进入生产环境，复杂度会立刻膨胀。

原因在于，真正的执行循环不只在做“推理 -> 调用工具 -> 再推理”，它还必须同时解决：

- 何时先规划，何时直接动手。
- 工具失败后是否重试。
- 哪些状态必须保留进下一轮。
- 什么情况下应该停下请求确认。
- 上下文变长之后如何继续。
- 复杂任务是否值得拆给子 Agent。

因此，执行循环的好坏，基本决定了一个 Agent 是“偶尔能跑通几步”，还是“能在复杂任务里持续推进很多步”。

## 13.2 Claude Code 的主循环，重点不在“会不会调工具”，而在“会不会继续”

围绕 `Claude Code` 的很多讨论，都喜欢把注意力放在它能调用多少工具、支持多少模式、能不能改代码上。

这些当然重要，但真正更有工程价值的，是它的主循环明显不是一次性问答结构，而是一种以“继续工作”为目标的运行方式。

这可以从前面几章已经看到的几个事实拼出来：

- 它会把 Prompt 做成动态装配层，而不是固定文案。
- 它把工具返回结果重新送回主循环，而不是一次调用后直接结束。
- 它把权限视为决策管道的一部分，而不是末端确认动作。
- 它承认长任务一定会触发压缩和恢复，而不是假装上下文永远够用。

这些点放在一起，说明 `Claude Code` 的执行循环目标不是“把当前一步做完”，而是“在多步任务里持续保持方向感”。

也正因为如此，执行循环对它来说不是局部实现细节，而是连接所有模块的运行时主干。

## 13.3 为什么复杂任务往往应该先规划

新补进来的参考资料里，有一条线索是值得保留的：`Claude Code` 的复杂任务处理明显比普通聊天系统更强调显式规划。

这条线索之所以重要，不是因为“规划”这个词听起来更高级，而是因为复杂代码任务的错误代价比普通问答高得多。

比如一个多步代码任务可能会涉及：

- 先定位相关文件。
- 再理解当前实现。
- 再决定修改边界。
- 再运行测试或命令。
- 再根据结果继续修正。

如果一开始方向就错了，后面的每一步都可能把错误放大。

因此，先规划的价值不在于让输出看起来更有条理，而在于给系统一个阶段性结构：

- 当前任务到底要完成什么。
- 哪些步骤是必要的。
- 哪些步骤存在高风险副作用。
- 哪些地方需要用户确认。

从工程上看，这相当于在正式执行之前，先把任务从“自然语言目标”转成“可继续追踪的执行框架”。

这也是为什么一个成熟执行循环里，规划不只是可选体验，而常常是降低错误放大概率的必要步骤。

## 13.4 好的执行循环必须能处理失败，而不是只处理成功路径

如果说规划解决的是“别一开始就走错”，那么错误恢复解决的就是“走错之后怎么办”。

这里是很多 Agent 系统最容易暴露真实成熟度的地方。

一个弱的执行循环通常会这样表现：

- 工具失败后，把错误原样塞回模型，看它自己怎么悟。
- 模型继续失败时，没有明显的熔断逻辑。
- 系统分不清“值得再试一次”和“应该换方案”。
- 连续几轮之后，用户已经看不出来它是在恢复，还是在打转。

而更成熟的执行循环，至少会在结构上承认几件事：

- 失败是常态，不是意外。
- 不同失败类型需要不同处理方式。
- 重试必须有边界。
- 长链路里必须防止错误恢复机制本身变成新的失控源。

这一点和前面的权限、压缩、恢复章节是连在一起的。因为长任务里的真正风险，往往不是第一次失败，而是系统在失败之后继续以不透明方式无限推进。

## 13.5 执行可见性为什么本身就是循环的一部分

很多人会把流式输出、中间步骤展示这类东西看成“产品体验优化”。这当然没错，但对 Agent 来说，它还有更深一层作用：执行可见性本身就是主循环的一部分。

原因很简单。

一旦系统开始多步行动，用户就不再只是结果接收者，而会变成循环中的潜在干预者：

- 用户可以中途叫停。
- 用户可以纠正方向。
- 用户可以补充约束。
- 用户可以在高风险步骤前拒绝继续。

因此，好的执行循环不是把一切藏在黑盒里跑完再汇报，而是让“用户观察和干预”天然地成为循环中的一个输入源。

这点看似偏产品，但其实直接影响架构。因为一套完全黑盒的执行循环，在复杂任务里往往更难治理，也更难建立信任。

## 13.6 子 Agent 不是花哨功能，而是上下文和任务管理手段

多 Agent 或子 Agent 很容易被讲成“更高级的智能协作”。这个说法并不完全错，但从工程角度看，子 Agent 首先解决的是两个更现实的问题：

第一，上下文污染。

复杂任务里，如果所有子任务都挤在同一个会话里，主上下文很快就会被大量局部信息塞满。这样做的结果通常不是“系统知道得更多”，而是“系统越来越抓不住重点”。

第二，串行瓶颈。

有些子任务其实相互独立，比如不同文件的调研、不同方案的比对、前后端分面分析、实现后再做审查。这类任务如果全部在单一循环里串行推进，会浪费大量时间和上下文注意力。

因此，子 Agent 的价值首先是任务管理和上下文治理：

- 把局部问题隔离出去。
- 让主循环保留更干净的工作台。
- 把可并行的工作拆开。
- 让结果以更浓缩的形式回流。

从这个角度讲，子 Agent 并不只是“让系统显得更聪明”，而是在帮助执行循环不要过早被复杂度压垮。

## 13.7 不同子 Agent 模式，本质上是在回答“要隔离到什么程度”

新增参考资料里提到了一些关于 `Claude Code` 子 Agent / teammate / worktree 的拆分线索。无论具体命名和实现细节后续怎样核验，这条工程问题本身是非常成立的：

一旦引入子 Agent，系统必须回答“子任务需要隔离到什么程度”。

通常至少有三种层次：

### 上下文分叉

子任务继承主任务的大部分背景，但在自己的上下文里局部展开。适合需要共享主线信息、但不想把细节持续污染主上下文的任务。

### 独立协作者

子任务有相对独立的上下文和执行路径，只把关键结论回传。适合相互依赖较低的并行工作。

### 文件系统或工作树隔离

子任务不仅上下文独立，连工作副本都独立，避免并发编辑互相覆盖。适合真正会改动同一仓库、但不应直接写进同一工作区的任务。

这几种隔离层次背后的问题其实非常朴素：

- 共享太少，子任务会缺信息。
- 共享太多，子任务会互相污染。
- 隔离太浅，文件改动会冲突。
- 隔离太深，协调成本又会变高。

所以，多 Agent 协作的真正难点，不是“能不能开多个 Agent”，而是“隔离和协调的粒度到底该画在哪里”。

## 13.8 Claude Code 这条线最值得社区学的，不是多 Agent 本身，而是怎么把它纳入主循环

这也是当前很多项目最容易做浅的地方。

不少系统把子 Agent 当成外挂能力：需要时随手起一个，跑完再塞回主任务。这样短期看起来够用，但结构上很容易出问题：

- 主循环不知道什么时候该拆任务。
- 子任务没有清楚的回流接口。
- 结果汇总靠大量自然语言堆叠。
- 并发任务一多，主循环本身反而更乱。

更值得借鉴的思路是：子 Agent 本身也应该被当成执行循环的一部分来设计。

这意味着系统要提前考虑：

- 什么类型的任务适合拆。
- 拆出去之后，子任务带走哪些上下文。
- 子任务返回什么形态的结果。
- 主循环如何在不重新吞下全部细节的情况下继续推进。

换句话说，子 Agent 不是执行循环之外的加速器，而是执行循环内部的复杂度管理手段。

## 13.9 从执行循环角度回看前面几章

如果用这一章的视角回看 `Claude Code` 前面的所有章节，会发现它们其实都在为执行循环服务：

- Prompt 装配解决的是每一轮推理前，系统到底给模型什么现场。
- 工具系统解决的是模型到底能以什么方式动手。
- 权限系统解决的是系统有没有资格继续推进。
- 记忆、压缩和恢复解决的是几十步之后系统还能不能保持主线。
- 子 Agent 解决的是复杂任务如何拆分、隔离和并行。

这也解释了为什么执行循环可以被称为 Agent 的“心脏”。因为前面每一层单独存在都没用，只有在循环里真正被接起来，系统才会变成一个可持续运行的代理。

## 13.10 对社区最有用的结论

把这章再压成几条最值得带走的结论，可以保留下面这些：

1. 执行循环是 Agent Runtime 最接近核心控制面的地方。
2. 好的执行循环重点不在“会不会调工具”，而在“会不会在多步任务里继续保持方向感”。
3. 规划、错误恢复、执行可见性和状态续航，是同一条主循环上的不同责任。
4. 子 Agent 的第一价值不是“显得更强”，而是管理上下文和任务复杂度。
5. 多 Agent 真正困难的不是数量，而是隔离粒度、回流接口和协调成本。

有了这一章，`Claude Code` 这条主线就不再只是几个模块的并列拆解，而开始呈现出更完整的运行时形态。
