# 前言：为什么要重新理解 Agent

过去一年，Agent 的叙事里混进了太多不必要的噪声。

一类噪声来自营销。一个项目只要能在终端里调几次工具、完成一个简单任务，就会被描述成“数字员工”“自治体”“下一代操作系统”。另一类噪声来自反向轻视。有人把 Agent 简化成“提示词套壳”“脚本编排”“工作流换皮”，仿佛除了模型以外没有任何新的工程问题。

这两种说法都不准确。

从工程视角看，Agent 最重要的部分既不是模型宣传，也不是某个单点技巧，而是运行时系统本身：它如何组织上下文，如何暴露工具，如何在不稳定模型之上建立可重复的执行行为，如何处理权限、安全、恢复、压缩和长期记忆，如何在一次次失败后仍保持边界清晰。

这也是本书选择以 `Claude Code`、`OpenClaw`、`Harness` 为主线的原因。

`Claude Code` 的价值，不只在于它是一个能改代码的 CLI，而在于它把本地代码代理做成了一个长期交互系统。启动链路、Prompt 装配、主循环、工具契约、权限决策、压缩和记忆，都已经不是零散功能，而是互相耦合的运行时设计。

`OpenClaw` 的价值，不只在于“能接微信、飞书、Telegram”，也不只在于技能系统，而在于它把“入口、大脑、执行器、扩展能力”拆成了更容易开源传播和二次搭建的形态。很多人真正想借鉴的，不是某个功能点，而是这套数字员工产品如何被工程化。

`Harness` 的价值，则在于它提供了一种更稳定的理解框架：模型提供能力上限，Harness 决定稳定性下限。一个 Agent 能不能持续工作，往往不取决于它一开始能回答多聪明，而取决于系统如何把上下文、工具、记忆、约束和反馈组织起来。

因此，这本书不打算写成三种东西：

- 不写成“功能大全”，因为功能目录解释不了系统为什么会这样设计。
- 不写成“神秘能力故事”，因为代码代理的价值主要来自工程，而不是神化模型。
- 不写成“纯产品宣传”，因为社区真正需要的是可复用经验，而不是二手情绪。

本书会优先回答四个问题：

1. Agent 运行时的主链路是什么。
2. 上下文、记忆、工具、权限各自负责什么。
3. 什么是值得复用的设计，什么只是阶段性实现。
4. 从 Claude Code、OpenClaw、Harness 身上，到底能沉淀出哪些最佳实践。

如果把这四个问题讲清楚，后面无论社区再出现什么新项目，它们都可以放回同一套分析框架里重新判断，而不必每次都从热闹开始。
