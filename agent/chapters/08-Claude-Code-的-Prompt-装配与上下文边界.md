# 第八章：Claude Code 的 Prompt 装配与上下文边界

前面已经提过，`Claude Code` 最不应该被误读成“一段很长的系统提示词”。如果只停在这个层面，很多关键工程判断都会被遮住。

更贴近事实的说法是：`Claude Code` 把 Prompt 做成了一层运行时装配系统。它关心的不是“提示词华不华丽”，而是当前这一轮到底该把哪些规则、哪些环境信息、哪些会话状态、哪些工具契约送进模型，以及哪些东西绝对不该常驻在上下文里。

这件事之所以重要，是因为 Agent 一旦进入真实环境，Prompt 处理的就不再只是“怎么提问”，而是“怎么组织整个可执行现场”。

## 8.1 Prompt 工程到这里已经变成上下文工程

新加入的参考文章里有一句话值得保留，但必须收敛地使用：`Agent` 很多时候是在用更多 `token` 和更高运行成本，换取更低的显式编排成本和更高的系统扩展性。

这个判断放在 `Claude Code` 上尤其合适。

传统脚本或硬编码工作流，会把大量控制流提前写死：何时读文件，何时查规则，何时换工具，何时重试，何时退出。Agent 不是不需要这些控制，而是把其中一部分迁移到了运行时，让模型在上下文约束内做判断。

一旦控制流被迁移到运行时，Prompt 就不再只是“给模型一个任务描述”，而必须承担更多职责：

- 告诉模型它当前处在什么系统里。
- 告诉模型有哪些长期规则和项目约束。
- 告诉模型本轮有哪些工具和行为边界。
- 告诉模型哪些信息是当前有效的，哪些需要自己再去读文件确认。

因此，Prompt 工程在这里自然演化成上下文工程。真正困难的部分，不是写一句聪明话，而是决定上下文里该放什么、不该放什么，以及放进去之后是否还能稳定复用。

## 8.2 `CLAUDE.md` 不是备忘录，而是分层规则面

Anthropic 官方文档已经把 `Claude Code` 的记忆位置讲得比较清楚。当前公开可确认的至少有四层：

- 企业级记忆：由组织集中下发。
- 项目级记忆：`./CLAUDE.md`。
- 用户级记忆：`~/.claude/CLAUDE.md`。
- 本地项目记忆：`./CLAUDE.local.md`，官方文档中已标注为逐步废弃，更推荐通过导入机制处理个人项目偏好。

这四层的工程意义，不只是“多几个文件路径”，而是明确了规则的作用域：

- 哪些规则属于组织治理，比如安全、合规、通用开发政策。
- 哪些规则属于项目团队共享，比如架构约定、测试命令、目录约束。
- 哪些规则属于个人偏好，比如命名习惯、常用命令、风格选择。
- 哪些规则只该在当前机器、当前项目生效。

这意味着 `CLAUDE.md` 并不是单纯的“让 Claude 记住一些事”，而是在给运行时提供一套分层规则面。规则面分得越清楚，系统越不容易因为用户临时补充一句要求就把项目约束冲掉。

## 8.3 递归发现与导入机制，说明它追求的是可组合性

官方文档还给出了两个很关键的机制：

第一，`Claude Code` 会从当前工作目录开始，向上递归查找 `CLAUDE.md`。这意味着在大型仓库里，不同层级可以自然叠加规则，而不是要求所有说明都挤在一个根文件里。

第二，`CLAUDE.md` 支持 `@path` 导入其他文件，且支持一定深度的递归导入。

这两个设计放在一起，说明 `Claude Code` 对上下文规则的处理方式不是“把所有说明堆在一个超长 markdown”，而是追求可组合性：

- 项目总览可以从 `README` 导入。
- 团队 git 流程可以放在独立文档里。
- 某个子目录的特殊约束可以只在进入该子树后再生效。
- 个人偏好可以通过家目录文件导入，而不污染团队仓库。

这其实已经非常接近配置系统，而不是传统意义上的注释文档。它的目标不是让人阅读舒服，而是让运行时能够稳定加载、分层覆盖、按目录生效。

## 8.4 上下文边界的关键不是“全放进去”，而是“按时进场”

很多早期 Agent 都容易犯一个错误：只要觉得有用，就尽量塞进系统上下文。

这在小 demo 里问题不大，但放到代码代理里会迅速失控。因为一个真实项目里永远存在三种信息：

- 长期稳定的规则。
- 当前任务阶段需要的环境信息。
- 一次性、局部性、只对某一步有效的临时材料。

如果三者不分层，结果通常是：

- 长期规则被短期噪声淹没。
- 局部材料长期滞留，持续污染后续推理。
- 模型开始根据旧状态行动，而不是根据当前状态行动。

这就是为什么前面说上下文更像工作台。工作台不是越大越好，而是越需要控制“哪些东西现在应该摆在桌面上”。

在 `Claude Code` 里，可以把进入上下文的内容粗略理解成三类：

- 启动即加载的长期规则，比如各层 `CLAUDE.md`。
- 会话内动态拼装的系统和环境信息。
- 在读文件、进子目录、执行工具之后临时纳入的任务材料。

真正重要的，不是三类内容是否都存在，而是它们不能混成一团。

## 8.5 为什么说 Prompt 装配本质上在做运行时预算管理

Prompt 装配还有一个经常被忽视的现实：它本质上也是预算管理。

这里的预算至少包括三种：

- `token` 预算。
- 模型注意力预算。
- 缓存稳定性预算。

第一种比较直观，上下文越长越贵。

第二种更关键。即使上下文还没达到窗口上限，模型也不可能对所有材料同样敏感。把大量低价值信息长期堆在前缀里，往往会让系统看起来“知道得更多”，实际却更容易抓不住重点。

第三种通常只有做过生产系统的人会特别在意。社区分析里提到，`Claude Code` 会区分更稳定的前缀和更动态的部分，并尽量维持某些部分的稳定顺序。这背后的目标，不只是整洁，而是提高缓存命中和上下文复用的稳定性。

所以从工程上看，Prompt 装配不是文案写作，而更像预算分配：

- 哪些规则值得常驻。
- 哪些内容应该延迟加载。
- 哪些信息只在读到特定路径时再引入。
- 哪些工具说明必须稳定排序。

当这些判断做得不好，模型看起来可能还是“能回答”，但长任务稳定性会明显下降。

## 8.6 Claude Code 在这里真正给社区上的一课

围绕 `Prompt` 和上下文，`Claude Code` 最值得社区借鉴的不是某个魔法模板，而是三条更硬的经验：

第一，规则必须分层。企业、项目、个人、目录局部约束，最好天然分开。

第二，上下文必须动态装配。不要把所有可能有用的信息长期塞在全局前缀里。

第三，Prompt 管理要像运行时模块一样设计，而不是像写文案一样维护。

这三条看似朴素，但它们直接决定了一个 Agent 到底是“偶尔灵一次”，还是“连续跑很多步仍然不太乱”。
